## 2. Wave 8 Code Quality Review

### 2a. `crates/llvm2-lower/src/dispatch.rs` -- Dispatch Codegen

**Verdict: GOOD**

- Clean architecture: `DispatchOp` enum covers all four dispatch cases (DataTransfer, KernelLaunch, Synchronize, CpuFallback).
- Topological sort via Kahn's algorithm is correct. Deterministic output via sorted queue.
- `generate_dispatch_plan()` properly handles:
  - Cross-target data transfers
  - Async target synchronization (GPU/ANE)
  - Final sync for pending async leaf nodes
  - Cost accumulation with `saturating_add`
- `validate_dispatch_plan()` checks all four required properties (assignment completeness, valid nodes, dependency satisfaction, sync-before-read).
- 15 well-structured tests cover scalar, NEON, GPU, ANE, mixed, and error cases.

**Minor issues:**
1. `topological_order()` uses `queue.pop()` (takes last element) but comments say "FIFO". The sort after each pop makes this deterministic regardless, but the comment is misleading. Not a bug -- just a documentation inconsistency.
2. The CPU fallback path at line 396-404 has a subtle condition: `target == CpuScalar && !assignment.contains_key(node_id)`. But `target` is already assigned `CpuScalar` as the default at line 357 when the node is missing from the assignment. So this fallback path only triggers when the node was NOT in recommendations AND defaults to CpuScalar. This is correct but could be clearer.

### 2b. `crates/llvm2-verify/src/synthesis.rs` -- CEGIS Upgrade

**Verdict: GOOD**

- Three verification modes (Evaluation, Cegis, EvaluationThenCegis) are well-designed with clear separation.
- The `EvaluationThenCegis` pipeline correctly uses evaluation as fast filter before CEGIS confirmation.
- CEGIS integration is clean: lazy initialization of `CegisLoop`, edge-case seeds.
- 44 tests with good coverage of positive/negative cases, mode routing, edge-case detection, backward compatibility.

**Minor issue:**
1. `verify_candidate_with_mode()` in `EvaluationThenCegis` mode increments `candidates_checked` in the Invalid/Unknown branches but also lets `verify_candidate_cegis()` increment it again in the Valid branch. This means a candidate that passes evaluation then goes to CEGIS gets counted twice. Not a functional bug but statistics will overcount.

### 2c. `crates/llvm2-ir/src/type_hierarchy.rs` -- Documentation Accuracy

**Verdict: GOOD**

- This is a pure documentation module (no code, no tests expected).
- The type hierarchy tables are accurate and match the actual type definitions in the codebase:
  - `MachFunction`, `MachInst`, `MachBlock`, `MachOperand`, `AArch64Opcode`, `InstFlags`, `VReg`, `PReg`, `RegClass`, `BlockId`, `InstId`, `StackSlotId` -- all verified present in `llvm2-ir`.
  - ISel types (`ISelFunction`, `ISelInst`, etc.) -- present in `llvm2-lower/isel.rs`.
  - RegAlloc types (`RegAllocFunction`, `RegAllocInst`, etc.) -- present in `llvm2-regalloc/machine_types.rs`.
- Pipeline flow diagram (Phases 1-9) matches the actual `pipeline.rs` implementation.
- Conversion method table is accurate and references correct source locations.

### 2d. `crates/llvm2-verify/src/unified_synthesis.rs` -- NEON Enumeration + Cost Integration

**Verdict: GOOD with minor warnings**

- NEON opcode enumeration is thorough: 13 opcodes (Add, Sub, Mul, Neg, And, Orr, Eor, Bic, Shl, Ushr, Sshr, Mla, Mls).
- Arrangement compatibility filter correctly excludes MUL/MLA/MLS from D2 (64-bit lane MUL not supported on AArch64 NEON).
- Cost model integration uses `llvm2_ir::cost_model` types (`CostModelGen`, `MultiTargetCostModel`, `ProfitabilityAnalyzer`).
- GPU/ANE candidate generation includes profitability gates.
- 67 tests covering scalar candidates, NEON candidates, cross-target search, cost ranking.

**Warnings:**
1. Unused imports in test code at line 2564: `CostModelGen` and `MultiTargetCostModel` are imported but not all used in that specific test function. These produce compiler warnings.
2. The `FullTarget` enum duplicates the `SynthTarget` enum with GPU/ANE additions. The comment explains this is to avoid circular dependencies, which is valid, but the `from_synth_target()` conversion adds a maintenance burden.

## 3. Pipeline Coherence

### Full Compilation Pipeline Trace

```
tMIR Function (llvm2-lower::Function)
  |
  v  Phase 1: ISel (llvm2-lower::isel)
ISelFunction
  |
  v  Phase 2: ISel -> IR (ISelFunction::to_ir_func())
MachFunction (llvm2-ir)
  |
  v  Phase 3: Optimization (llvm2-opt)
MachFunction (optimized)
  |
  v  Phase 4: IR -> RegAlloc (pipeline::ir_to_regalloc())
RegAllocFunction (llvm2-regalloc)
  |
  v  Phase 5: Register Allocation
AllocationResult (VReg -> PReg map)
  |
  v  Phase 6: Apply allocation
MachFunction (PRegs only)
  |
  v  Phase 7: Frame Lowering (llvm2-codegen::frame)
MachFunction (with prologue/epilogue)
  |
  v  Phase 8: Encoding (llvm2-codegen::aarch64)
Vec<u8> (machine code bytes)
  |
  v  Phase 9: Mach-O Emission (llvm2-codegen::macho)
Object file bytes
```

**Pipeline handoffs verified:**
- Phase 1->2: `ISelFunction::to_ir_func()` exists and is tested in isel.rs
- Phase 2->3: llvm2-opt works directly on `MachFunction` (correct)
- Phase 3->4: `pipeline::ir_to_regalloc()` exists with proper `From`/`TryFrom` adapters
- Phase 4->5: RegAllocFunction feeds into regalloc (correct)
- Phase 5->6: AllocationResult applied back to MachFunction (correct)
- Phase 7-9: Codegen works on MachFunction directly (correct)

**Verification pipeline (parallel track):**
- `llvm2-verify` is invoked separately, not inline in the compilation pipeline
- This is correct for now (verification is optional/feature-gated)
- The synthesis engine (`unified_synthesis.rs`) feeds discovered rules back to the optimization pass manager

### Dangling Modules

**No dangling modules found.** All modules in each crate's `lib.rs` are reachable:
- Every `.rs` file is declared as `pub mod` in its crate's `lib.rs` or parent `mod.rs`
- The `llvm2-lower/src/function.rs` and `llvm2-lower/src/instructions.rs` modules define core types used throughout the crate
- The `llvm2-codegen/src/pipeline.rs` module has 0 tests but is the central pipeline orchestrator -- it is exercised via the `compiler.rs` integration tests

One dead code item detected by the compiler: `read_array_elem_f64` in `ane_semantics.rs` (line 663). This is an unused helper function that should be removed or used.
