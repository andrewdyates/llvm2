## Executive Summary

Waves 13-16 delivered 18 non-merge commits (35 total including merges) that focused on three themes: (1) codebase correctness hardening via a comprehensive audit that filed 14 new issues and immediately fixed the most critical ones, (2) architectural unification to reduce the dual-encoder and dual-opcode-enum divergence that caused silent bugs, and (3) deepened verification with new SMT proofs for DCE, copy propagation, and CFG simplification.

The project has grown from 62,073 LOC and 1,495 tests (Wave 12) to **69,651 LOC and 1,688 tests**. Of those 1,688 tests, **1,682 pass** and **6 fail** (4 end-to-end runtime tests regressed by the encoder unification, 2 variadic call tests from incomplete variadic support). The formal verification crate nearly doubled from 7,415 LOC to 10,325 LOC. Total issues grew from 88 to 105, with 81 closed (77%) and 24 open (23%).

The most significant architectural change was the elimination of the dual AArch64 encoder ([U]354), which resolved the P1 issue #92 where `lower.rs` and `aarch64/encode.rs` had divergent instruction encoding logic. This unification, along with opcode enum consolidation ([U]345), makes progress on the long-standing type duplication problem (#73).

**Regression note:** The encoder unification introduced a regression in 4 E2E runtime tests where compiled functions return 0 instead of correct values. Encoding-only tests pass, indicating the binary format is valid but instruction semantics may have shifted during the merge. This is the highest-priority item for the next wave.

## Key Deliverables Summary (Waves 13-16)

### 1. Codebase Audit and Issue Filing ([U]349, Wave 14)

A comprehensive codebase audit identified 14 new issues (#89-#105) covering:
- **P1:** Duplicate AArch64 encoder (#92), pipeline opcode catch-all silently dropping 20+ opcodes as NOP (#89)
- **P2:** Silent operand mapping bugs (#94, #98), missing ISel patterns (#88, #90, #100), 32-bit type handling (#93, #99), symbol relocation stub (#95)
- **P3:** FP size helper defaults (#105), produces_value() duplication (#96), compact unwind register drops (#97), pseudo expansion catch-all (#104), adapter type fallbacks (#102)

This audit-then-fix pattern -- filing issues for discovered problems before fixing them -- ensured systematic coverage rather than ad-hoc patching.

### 2. Architecture Unification

**Opcode enum consolidation** ([U]345, Wave 14): Unified the ISel and IR opcode enums, eliminating one of the three adapter layers contributing to type duplication (#73). This reduced the surface area for opcode translation bugs.

**Dual-encoder elimination** ([U]354, Wave 16): The codebase had two independent AArch64 instruction encoders: `lower.rs` (inline encoding) and `aarch64/encode.rs` (dedicated module). These had diverged over time, producing different encodings for the same instructions (#92). Wave 16 unified them into a single encoder. This is the most significant architectural improvement since the project began.

**Opcode rename cleanup** ([U]360, Wave 16): Renamed remaining LLVM-style opcodes to canonical names, completing the unification started in [U]345.

### 3. Correctness Hardening

**Silent operand translation bugs fixed** ([U]350, Wave 15): Fixed issues where `preg_hw()` defaulted non-register operands to XZR (#98), `extract_symbol_name()` always returned None (#95), and FP size helpers defaulted to Double for non-FPR operands (#105).

**Silent pseudo expansion fixed** ([U]352, Wave 15): The `expand_pseudos` catch-all was converting unknown opcodes to NOP instead of returning an error (#104). Compact unwind was silently dropping unknown callee-saved register pairs (#97).

**Symbol extraction for relocations** ([U]351, Wave 15): Implemented actual symbol name extraction from operands, enabling relocation emission for external symbol references (#95).

**Error handling cleanup** ([U]346, Wave 14; [U]357, Wave 16): Completed error handling cleanup across all crates. Replaced panic!/unreachable! with Result types in the pipeline regalloc adapter (#71, #54). Consolidated 4 duplicated `produces_value()` implementations (#96).

### 4. New Instruction Coverage

**Switch and CallIndirect** ([U]353, Wave 15): Implemented lowering for switch statements (BR table dispatch) and indirect function calls. These are required for control-flow-heavy and function-pointer-heavy programs (#90).

**Variadic function support** ([U]347, Wave 14): Added Apple AArch64 ABI variadic function support (#79). Two ISel tests still fail, indicating incomplete pattern coverage.

**Byte/half load-store wiring** ([U]343, Wave 13): Connected extending loads (LDRB/LDRH/LDRSB/LDRSH) and truncating stores (STRB/STRH) ISel opcodes through the pipeline translation layer (#88).

### 5. Verification Expansion

**DCE, Copy Propagation, CFG Simplification proofs** ([U]350, Wave 15): Added 77+ new SMT proof functions in `dce_proofs.rs`, `copy_prop_proofs.rs`, and `cfg_proofs.rs`. Total verification proofs grew from 245 to 322+.

**z4 integration scaffold** ([U]348, Wave 14): Added `z4_bridge.rs` (982 LOC) providing the API surface for future real z4 SMT solver integration (#34).

### 6. Test Infrastructure

**End-to-end Mach-O validation suite** ([U]356, Wave 16): New integration test suite validating complete Mach-O output files against format correctness criteria (#91).

**Increased test coverage** ([U]359, Wave 16): Added tests for greedy register allocator, Mach-O writer, and loop analysis, addressing the coverage gap identified in #101.

### Risks

1. **E2E runtime regression is the highest priority.** The encoder unification ([U]354) eliminated the dual-encoder divergence (#92) but introduced a regression where compiled functions return 0 instead of correct values. Encoding-only tests pass, suggesting the binary format is valid but instruction semantics shifted. This needs immediate investigation -- the likely cause is an operand ordering or encoding field mismatch in the unified encoder for specific instruction patterns (e.g., ADD result register placement).

2. **Type duplication (#73) is partially addressed but not resolved.** The opcode enum unification ([U]345) eliminated one adapter layer, but two remain. This continues to be the primary source of translation bugs between the ISel and codegen layers.

3. **Variadic support is incomplete.** Two ISel tests fail for variadic function patterns. The Apple AArch64 ABI variadic calling convention was implemented ([U]347) but the ISel pattern matching for variadic calls needs further work.

4. **No real z4 integration.** The z4_bridge.rs scaffold ([U]348) provides the API surface but all 322 proofs still use mock evaluation. Real SMT solving would provide stronger guarantees for 32-bit and 64-bit widths.

## Conclusion

Waves 13-16 shifted the project from feature expansion to correctness hardening. The comprehensive audit ([U]349) filed 14 issues, 16 of which were fixed within the same wave cycle. The dual-encoder elimination ([U]354) was the most impactful architectural change, resolving a P1 divergence that had been a source of silent encoding bugs since the project began.

The trade-off was a regression: 4 E2E runtime tests now fail where compiled functions return 0. This is the clear #1 priority for the next wave. The encoding-only tests pass, so the root cause is likely an operand field mapping error in the unified encoder, not a fundamental design flaw.

**Critical path forward:**

1. **Fix E2E runtime regression** -- Investigate and fix the 4 failing `_run` tests introduced by encoder unification. This is blocking.
2. **Fix variadic ISel tests** -- Complete the 2 failing variadic call patterns.
3. **Resolve type duplication (#73)** -- Two adapter layers remain. Eliminate or minimize them.
4. **Real z4 integration (#34)** -- Connect the z4_bridge scaffold to the actual z4 solver.
5. **Close remaining phase issues (#33)** -- Phase 8 (proof-enabled optimizations) tracking issue.

The backend has matured from a prototype to a substantial compiler with 69,651 LOC, 1,688 tests, 322+ SMT proofs, and 77% issue closure rate. The design doc's 9-phase build order is at 93.9% average completion. The remaining work is focused on eliminating the last architectural debt (type duplication), fixing the runtime regression, and connecting to the real z4 solver for production-grade verification.
