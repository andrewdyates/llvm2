## Findings

### P1: `is_64bit` helper in lower.rs always returns true for GPRs (wrong sf for W-registers)

**File:** `crates/llvm2-codegen/src/lower.rs:256-260`

```rust
let is_64bit = |idx: usize| -> bool {
    match inst.operands.get(idx) {
        Some(MachOperand::PReg(p)) => p.is_gpr() && p.encoding() < 32,
        _ => true,
    }
};
```

**Bug:** `PReg::is_gpr()` only returns true for encodings 0-31 (X registers / Gpr64). W registers have encodings 32-63 and `is_gpr()` returns false for them, so the `_ => true` default fires, making sf=1 (64-bit). This means:

- **BicRR, Csinc, Csinv, Csneg, Movn** with W-register operands will encode with sf=1 instead of sf=0
- These are opcodes handled locally in lower.rs (not delegated to encode.rs)
- The unified encoder (encode.rs) correctly uses `sf_from_operand()` which checks `preg_class()` and correctly maps Gpr32 -> sf=0

**Impact:** Miscompilation for 32-bit conditional select, bit-clear, and move-not operations. Currently latent because ISel likely always produces X-register operands for these, but any future 32-bit usage will silently generate wrong code.

**Fix:** Replace `is_64bit` with the same `preg_class` approach used by encode.rs:
```rust
let is_64bit = |idx: usize| -> bool {
    match inst.operands.get(idx) {
        Some(MachOperand::PReg(p)) => {
            matches!(preg_class(*p), RegClass::Gpr64)
        }
        Some(MachOperand::Special(SpecialReg::WZR)) => false,
        _ => true,
    }
};
```

### P1: encode.rs `preg_hw` silently defaults non-register operands to XZR (reg 31)

**File:** `crates/llvm2-codegen/src/aarch64/encode.rs:62-69`

```rust
fn preg_hw(inst: &MachInst, idx: usize) -> u32 {
    match inst.operands.get(idx) {
        Some(MachOperand::PReg(p)) => p.hw_enc() as u32,
        Some(MachOperand::Special(s)) => match s {
            SpecialReg::SP | SpecialReg::XZR | SpecialReg::WZR => 31,
        },
        _ => 31, // silent default!
    }
}
```

lower.rs explicitly validates register operands and returns errors for non-register operands (see bug #98 comments at lower.rs:222-238). But if any opcode falls through the `_ =>` default match in lower.rs's `encode_inst` to the unified encoder, and the instruction has a malformed operand (e.g., Imm where a register is expected), encode.rs will silently encode it as XZR. lower.rs has pre-validation for all known opcodes, but any newly added opcode that falls to the default path will lose this safety net.

**Impact:** Silent miscompilation for malformed instructions reaching the unified encoder without lower.rs pre-validation. Currently mitigated by lower.rs's explicit validation of known opcodes.

### P2: BicRR, Csinc, Csinv, Csneg, Movn duplicated between lower.rs and encode.rs

**Files:** 
- `crates/llvm2-codegen/src/lower.rs:324-384` (local implementations)
- `crates/llvm2-codegen/src/aarch64/encode.rs:1309-1320` (listed as UnsupportedOpcode)

These five opcodes exist in lower.rs with local encoding implementations AND are listed in encode.rs's UnsupportedOpcode block. This is documented and intentional (lower.rs comments at lines 207-210), but creates maintenance risk:

1. If encode.rs later implements these opcodes, lower.rs's local versions won't be removed automatically
2. The local implementations have the `is_64bit` bug (P1 above) while encode.rs would presumably use `sf_from_operand`
3. AndRI, OrrRI, EorRI, FmovImm are also in the UnsupportedOpcode list -- these have no local implementation anywhere

**Recommendation:** Track in issue #73 that unification is pending for these 5+4 opcodes.

### P2: NEON lowering proofs only test specific shift amounts, not boundary values

**File:** `crates/llvm2-verify/src/neon_lowering_proofs.rs:414-421, 448-453, 479-484`

The shift proofs use hardcoded moderate shift amounts:
- SHL.4H #4, SHL.4S #8
- USHR.8B #2, USHR.2D #16
- SSHR.2S #1, SSHR.8H #4

These are all valid but do not test boundary conditions:
- Shift by 0 (identity)
- Shift by lane_width - 1 (maximum valid shift)
- These are the most likely values to trigger encoding bugs in the `h:chl` encoding of the NEON immediate shift field

The proofs themselves are structurally correct (tMIR semantics match NEON semantics via `map_lanes_binary_imm` / `bvshl` / `bvlshr` / `bvashr`), but the coverage is narrow.

### P2: NEON proofs have no preconditions on shift amounts

**File:** `crates/llvm2-verify/src/neon_lowering_proofs.rs:398-410`

All NEON shift proof obligations have `preconditions: vec![]`. In real hardware, NEON shift amounts must be in range [0, lane_width). An out-of-range shift would produce undefined behavior. The proofs should include a precondition constraining the shift amount, or document why it's safe to omit (answer: the shift is a compile-time constant embedded in the instruction, not a runtime variable, so ISel is responsible for range-checking).

### P3: `proof_guided_partition_cost()` defined and tested but no callers outside tests

**File:** `crates/llvm2-lower/src/compute_graph.rs:391-408`

The `proof_guided_partition_cost()` method is implemented and has 3 test functions exercising it, but a grep across the codebase finds no production callers. Similarly, `target_recommendations()` and `annotate_with_proofs()` are only called from tests.

This is expected at this stage (the heterogeneous compute pipeline is Phase 1 only), but should be tracked to ensure these methods get integrated into the actual compilation pipeline.

### P3: Lower Type enum missing `Ptr` variant present in IR Type

**Files:**
- `crates/llvm2-lower/src/types.rs:33` (no `Ptr` variant)
- `crates/llvm2-ir/src/function.rs:57` (has `Ptr` variant)

The `From<Type> for llvm2_ir::function::Type` impl at `types.rs:176-197` does not handle `Ptr` because lower's Type enum doesn't have it. This is documented in the comment at line 170-171: "The llvm2-ir Type has a `Ptr` variant that llvm2-lower's Type lacks; this conversion never produces Ptr."

This is acceptable for now but means pointer-typed values flowing through the lowering pipeline will need special handling when pointer support is implemented.

### P3: `imm_val` helper in lower.rs silently returns 0 for non-immediate operands

**File:** `crates/llvm2-codegen/src/lower.rs:242-253`

```rust
let imm_val = |idx: usize| -> i64 {
    inst.operands
        .get(idx)
        .and_then(|op| {
            if let MachOperand::Imm(v) = op {
                Some(*v)
            } else {
                None
            }
        })
        .unwrap_or(0)
};
```

Silently returning 0 for missing or wrong-type operands masks bugs. While `preg_hw` was fixed (bug #98) to return errors, `imm_val` was not. Used for condition codes in Csinc/Csinv/Csneg where a wrong 0 would mean "EQ condition" -- potentially wrong but not catastrophic.

- MOVWrr, MOVXrr, STRWui, STRXui, STRSui, STRDui, BL, BLR, CMPWrr, CMPXrr, CMPWri, CMPXri, MOVZWi, MOVZXi, Bcc (13 typed aliases)

### Assessment
The encoder unification is solid. All production opcodes either delegate to encode.rs or have documented reasons for staying local. The only gap is the 5 locally-encoded opcodes that should migrate to encode.rs when #73 is completed.

## NEON Proofs Assessment

The 22 NEON lowering proofs (11 ops x 2 arrangements) are structurally sound:
- tMIR side correctly uses `map_lanes_binary` / `map_lanes_unary` / `map_lanes_binary_imm`
- NEON side uses dedicated encode functions from `neon_semantics.rs`
- All proofs pass verification via mock evaluation
- Negative test confirms wrong lowering is detected

Arrangements are correctly encoded (lane counts and element sizes match ARM DDI 0487). The only gaps are shift boundary testing (P2) and missing preconditions documentation (P2).
