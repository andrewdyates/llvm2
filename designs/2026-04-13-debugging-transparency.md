## Introduction

Traditional compilers are black boxes. Ask "why did clang emit this instruction?" and you'll get silence. LLVM has opt-remarks (`-Rpass`) that tell you *which* optimization fired, but not *why* it was chosen, what alternatives were considered, or how to trace an output instruction back to its source expression.

LLVM2 is a **glass box compiler**. Every compilation decision is logged, justified, and traceable — from tMIR input to final binary. Proof certificates ship alongside the binary. An interactive explorer lets you query any instruction.

In an AI era where most code is generated by AI, transparency is non-negotiable. You must be able to audit the compilation chain.

## Prior Art

### LLVM Optimization Remarks

**System:** `-Rpass=<regex>`, `-Rpass-missed`, `-Rpass-analysis` emit structured YAML records. Each remark: pass name, function, source location (file:line:col), hotness (from PGO), human-readable message. Three categories: passed (applied), missed (tried but failed), analysis (informational).

**Tools:** `opt-viewer.py` generates per-source-file HTML with inline annotations. `opt-stats.py` aggregates. `opt-diff.py` compares.

**Answers:** "Why wasn't this loop vectorized?" "What inlining decisions were made?"

**Cannot answer:** "Which source expression produced this specific machine instruction." Remarks are pass-centric, not instruction-centric. No "why" — only "what happened."

Ref: https://llvm.org/docs/Remarks.html

### LLVM opt-bisect

`-opt-bisect-limit=N` runs only the first N optimization passes. Binary search isolates which pass introduced a regression. `-opt-bisect-print-ir-path` dumps IR at the bisection point.

**Limitation:** Granularity is per-pass-invocation, not per-transformation. Analysis passes can't be skipped.

Ref: https://llvm.org/docs/OptBisect.html

### GCC -fopt-info

Four categories: optimized, missed, note, all. Filterable by group (inline, vec, loop). Unstructured text to stderr.

**Limitation:** No structured format, no visualization, no hotness. Inferior to LLVM remarks.

### CompCert Verification Certificates

Each pass (C → Clight → Cminor → RTL → LTL → Linear → Mach → asm) has a paired Coq proof. Proofs use simulation relations: for every source step, the target takes corresponding steps preserving observable behavior.

**Limitation:** Proofs are in Coq, not runtime certificates. Users can't query "why was this emitted." Proofs prove existence of correctness, not explain decisions.

Ref: https://compcert.org/doc/

### Alive2

Encodes LLVM IR (source + target) as SMT (Z3), checks refinement. On failure, produces concrete counterexample with divergent input/output values. Web UI, standalone tool, drop-in compiler wrappers.

**Limitation:** Intraprocedural only.

Ref: https://github.com/AliveToolkit/alive2

### Source Location Tracking (DILocation)

Every LLVM IR instruction carries source location. Passes must: preserve for block-local transforms, merge via `applyMergedLocation()`, drop via `dropLocation()` when ambiguous. `debugify` pass inserts synthetic locations and verifies they survive.

**Limitation:** Designed for debuggers, not compilation explanation. Merged/dropped locations lose the "why."

## Gap Analysis

No existing system provides full instruction-level provenance:

| Capability | LLVM | CompCert | Alive2 | LLVM2 (goal) |
|-----------|------|----------|--------|--------------|
| Pass-level decision logging | Remarks | N/A | N/A | **Full event log** |
| Pass-level bisection | opt-bisect | N/A | N/A | **Automated bisection** |
| Source-to-instruction mapping | DILocation | Simulation | N/A | **Full provenance** |
| Per-rule correctness proof | N/A | Coq (static) | SMT per-transform | **Runtime certificates** |
| Counterexample on failure | N/A | N/A | Concrete values | **With provenance** |
| Decision justification | N/A | N/A | N/A | **WHY, not just WHAT** |

LLVM2's opportunity: unify all columns. Each lowering rule emits a structured record: source tMIR instruction, emitted machine instruction(s), rule name, z4 proof obligation, cost model justification.

## LLVM2 Transparency Architecture

### 1. Compilation Event Log

Every pass emits structured events to a `CompilationTrace`:

```rust
pub struct CompilationEvent {
    /// Which pass emitted this event
    pub pass: PassId,
    /// What happened
    pub kind: EventKind,
    /// The instruction(s) affected
    pub subject: Vec<InstId>,
    /// Why this decision was made
    pub justification: Justification,
    /// Timestamp (monotonic, for ordering)
    pub seq: u64,
}

pub enum EventKind {
    /// Optimization applied
    Applied { rule: RuleId, before: Vec<InstId>, after: Vec<InstId> },
    /// Optimization considered but rejected
    Rejected { rule: RuleId, reason: String },
    /// Instruction lowered from tMIR to MachIR
    Lowered { tmir_inst: TmirInstId, mach_insts: Vec<InstId> },
    /// Register assigned
    RegAssigned { vreg: VRegId, preg: PReg, reason: String },
    /// Value spilled to stack
    Spilled { vreg: VRegId, slot: StackSlot, cost: f64 },
    /// Instruction encoded to binary
    Encoded { inst: InstId, offset: u32, bytes: Vec<u8> },
}

pub enum Justification {
    /// Cost model says this is better
    CostModel { before: f64, after: f64 },
    /// Pattern match triggered
    PatternMatch { pattern: String },
    /// SMT solver proved equivalence
    SolverProved { proof_hash: u64 },
    /// Legality constraint
    Legality { constraint: String },
    /// Profile data
    ProfileGuided { hotness: u64 },
}
```

**Opt-in levels:**
- `Level::None` — no logging (production default)
- `Level::Summary` — pass-level applied/rejected counts
- `Level::Full` — every event with justification
- `Level::Debug` — full + intermediate IR snapshots

### 2. Provenance Tracking

Source-to-binary mapping maintained through every transformation:

```rust
pub struct ProvenanceMap {
    /// tMIR instruction → MachIR instructions (1:N from ISel)
    tmir_to_mach: HashMap<TmirInstId, Vec<InstId>>,
    /// MachIR instruction → binary offset (1:1 from encoding)
    mach_to_offset: HashMap<InstId, u32>,
    /// Transitive: tMIR → binary offset
    tmir_to_offset: HashMap<TmirInstId, Vec<u32>>,
}
```

**Rules for passes:**
- **ISel:** Creates initial mapping (tMIR → MachIR)
- **Optimization passes:** When replacing inst A with inst B, update mapping: `provenance[B] = provenance[A]`
- **When merging instructions:** Record both sources
- **When deleting (DCE):** Mark as "optimized away" with justification
- **Encoding:** Adds final offset mapping

### 3. Proof Certificates

Portable artifacts shipped alongside the binary:

```
binary.o          — the compiled object file
binary.proof.cbor — proof certificate (CBOR-encoded)
```

**Certificate contents:**
- For each lowering rule applied: the z4 proof obligation and result
- For each optimization: the equivalence proof
- For each register assignment: legality proof (no conflicts)
- Machine-verifiable: an independent checker can validate without the compiler

**Format:** CBOR (Concise Binary Object Representation) — compact, well-specified, language-independent. JSON fallback for human inspection.

### 4. Decision Justification

Every heuristic decision records WHY:

```rust
pub trait JustifiedDecision {
    /// The decision made
    fn decision(&self) -> &str;
    /// Why this was chosen over alternatives
    fn justification(&self) -> Justification;
    /// What alternatives were considered
    fn alternatives(&self) -> Vec<(String, Justification)>;
}
```

Examples:
- "Inlined function `foo` because: callee is 3 instructions, call overhead is 4 instructions, net savings = 1 instruction"
- "Did NOT vectorize loop at L42 because: trip count unknown, would require runtime check costing 5 instructions"
- "Assigned VReg %7 to X3 because: X0-X2 occupied by live values, X3 is callee-saved (no save/restore needed since function already saves X3)"

### 5. Regression Bisection

Automated tool to find which pass degraded output:

```bash
llvm2-bisect --metric=code-size --threshold=5% input.tmir
```

Algorithm:
1. Run full pipeline, measure metric (code size, estimated cycles)
2. Binary search: disable passes, find the one that causes regression
3. Report: "Pass `peephole` at position 7/11 increased code size by 12 bytes. Before/after IR diff attached."

### 6. Output Formats

| Format | Use |
|--------|-----|
| JSON compilation trace | Programmatic consumption by AI agents |
| CBOR proof certificates | Compact, machine-verifiable proofs |
| SARIF diagnostics | IDE integration (VS Code, JetBrains) |
| HTML report | Human-readable compilation explorer |
| Binary IR dumps | Exact IR state at each pipeline stage |

## Interactive Compilation Explorer

### CLI Tool

```bash
# Query a specific output instruction
llvm2-explain binary.o --offset=0x48
# Output:
# Instruction at 0x48: ADD X0, X1, X2
# Source: tMIR %5 = add i64 %3, %4  (foo.rs:42:5)
# Lowered by: ISel rule "i64_add_rr" (proof: abc123)
# Survived passes: DCE (kept: has users), CopyProp (unchanged)
# Register assignment: %3→X1 (live range [12,28]), %4→X2 (live range [8,28])
# Cost: 1 cycle latency, 0.25 reciprocal throughput

# Show full transformation chain
llvm2-explain binary.o --offset=0x48 --chain
# Output shows IR at each stage with the instruction highlighted

# Compare before/after a specific pass
llvm2-explain binary.o --pass=peephole --diff
```

### Structured API

```rust
pub struct CompilationExplorer {
    trace: CompilationTrace,
    provenance: ProvenanceMap,
    certificates: ProofCertificates,
}

impl CompilationExplorer {
    /// Query instruction at binary offset
    pub fn explain_offset(&self, offset: u32) -> InstructionExplanation;
    
    /// Show transformation chain for a tMIR instruction
    pub fn trace_instruction(&self, tmir_id: TmirInstId) -> TransformationChain;
    
    /// Diff IR before/after a pass
    pub fn pass_diff(&self, pass: PassId) -> IrDiff;
    
    /// Find which pass affected a metric
    pub fn bisect_metric(&self, metric: Metric) -> BisectionResult;
}
```

## Implementation Plan

### Phase 1: Event Log Infrastructure
- `CompilationEvent` struct and `CompilationTrace` collector
- Emit events from ISel (lowering events) and encoding (binary offset events)
- JSON serialization

### Phase 2: Provenance Tracking
- ProvenanceMap through ISel
- Update rules for optimization passes
- tMIR-to-binary transitive mapping

### Phase 3: Proof Certificates
- CBOR serialization of z4 proof obligations and results
- Independent certificate checker tool
- Ship alongside .o files

### Phase 4: Decision Justification
- JustifiedDecision trait
- Implement for register allocation, peephole, ISel
- "Why" logging for every heuristic

### Phase 5: Explorer and Bisection
- `llvm2-explain` CLI tool
- HTML report generation
- Automated pass bisection
