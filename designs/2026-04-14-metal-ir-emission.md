# Metal IR Emission: GPU Targeting via AIR Generation

**Author:** Andrew Yates
**Date:** 2026-04-14

---

## Problem

The GPU semantic encoding (`gpu_semantics.rs`) and dispatch plan generator (`dispatch.rs`) prove that a computation is correct on a Metal GPU and decide when to dispatch to it. What remains is the **emission** step: generating actual Metal compute kernels and the host-side code that launches them. This document specifies how LLVM2 emits Metal IR (Apple Intermediate Representation, "AIR") and integrates with the Metal runtime for GPU execution.

---

## Metal Compilation Pipeline

Apple's Metal shader compilation has two well-defined stages:

```
MSL source code                      AIR bitcode
  (.metal)                            (.air)
      |                                 |
      v                                 v
metal frontend                   metallib archiver
(xcrun -sdk macosx metal)       (xcrun -sdk macosx metallib)
      |                                 |
      v                                 v
   AIR bitcode (.air)             Metal Library (.metallib)
                                        |
                                        v
                                 GPU driver JIT
                                 (at pipeline creation time)
                                        |
                                        v
                                 Native GPU ISA
```

**Ref:** Apple. "Building a Shader Library by Precompiling Shader Source." Apple Developer Documentation, 2024.

### Two Emission Strategies

| Strategy | Input | Tool | Output | When |
|----------|-------|------|--------|------|
| **MSL Source** | `.metal` text | `metal` compiler | `.air` bitcode | Simpler, human-readable, slower |
| **AIR Direct** | LLVM bitcode (AIR dialect) | `metallib` | `.metallib` | Faster, no parse overhead, lower-level |

LLVM2 should support both strategies:

- **Phase 1 (MVP):** Emit MSL source text, compile via `xcrun metal`. Simplest to implement, easiest to debug, human-readable output.
- **Phase 2 (Optimized):** Emit AIR bitcode directly. AIR is a dialect of LLVM IR with Metal-specific intrinsics. This bypasses the Metal frontend parser entirely.

This design focuses on Phase 1 (MSL emission) with notes on the Phase 2 AIR path.

## AIR (Apple Intermediate Representation) Format

AIR is LLVM IR bitcode with Metal-specific extensions. Key properties:

| Property | Detail |
|----------|--------|
| Base format | LLVM bitcode (`.bc` container) |
| Target triple | `air64-apple-macosx` or `air64-apple-ios` |
| Address spaces | 0=device, 1=constant, 2=threadgroup, 3=thread |
| Intrinsics | `@air.get_thread_position_in_grid`, `@air.get_threadgroup_position_in_grid`, etc. |
| Attributes | `[[buffer(N)]]` maps to `!metal.buffer` metadata |
| SIMD ops | `@air.simd_sum`, `@air.simd_shuffle`, `@air.simdgroup_matrix_*` |

**Ref:** Apple. "Metal Shading Language Specification," Version 3.2, Appendix A: Metal IR.

### Metal Address Spaces

```
Address Space 0: device     — GPU global memory (MTLBuffer with storageModeShared on UMA)
Address Space 1: constant   — Read-only data (optimized caching)
Address Space 2: threadgroup — Shared memory within a threadgroup (32KB max)
Address Space 3: thread     — Per-thread private memory (registers/stack)
```

These map directly to the data flow in our dispatch plan. Input buffers use address space 0 (device) or 1 (constant). Threadgroup temporaries use address space 2. Per-thread accumulators use address space 3.

## MSL Source Emission (Phase 1)

### Kernel Template Structure

LLVM2 emits Metal Shading Language source code for each GPU kernel in the dispatch plan. Each kernel follows a consistent template:

```metal
#include <metal_stdlib>
using namespace metal;

// Kernel: <descriptive_name>
// Generated by LLVM2 — verified correct via z4 proof
// Source: tMIR function <func_name>, compute node <node_id>
// Pattern: <ParallelMap|ParallelReduce|MapReduce|MatMul>

kernel void llvm2_kernel_<node_id>(
    const device <elem_type>* input  [[buffer(0)]],
    device <elem_type>* output       [[buffer(1)]],
    uint tid [[thread_position_in_grid]],
    uint tgid [[threadgroup_position_in_grid]],
    uint lid [[thread_position_in_threadgroup]])
{
    // Bounds check: grid may be larger than data
    if (tid >= <grid_size>) return;

    // <kernel body — depends on pattern>
}
```

### Pattern-Specific Kernel Bodies

#### Parallel Map

Maps directly from `GpuKernelPattern::ParallelMap` in the semantic encoding.

```metal
// Element-wise: output[tid] = f(input[tid])
kernel void llvm2_map_<node_id>(
    const device float* input  [[buffer(0)]],
    device float* output       [[buffer(1)]],
    uint tid [[thread_position_in_grid]])
{
    if (tid >= N) return;
    output[tid] = <f(input[tid])>;
}
```

For binary maps (two input arrays):

```metal
kernel void llvm2_map2_<node_id>(
    const device float* a [[buffer(0)]],
    const device float* b [[buffer(1)]],
    device float* output  [[buffer(2)]],
    uint tid [[thread_position_in_grid]])
{
    if (tid >= N) return;
    output[tid] = <f(a[tid], b[tid])>;
}
```

The per-element function `f` is emitted as an inline MSL expression. The expression generator maps tMIR operations to MSL operators:

| tMIR Op | MSL Expression |
|---------|---------------|
| `Add(a, b)` | `a + b` |
| `Sub(a, b)` | `a - b` |
| `Mul(a, b)` | `a * b` |
| `Div(a, b)` | `a / b` |
| `Neg(a)` | `-a` |
| `Abs(a)` | `abs(a)` |
| `Sqrt(a)` | `sqrt(a)` |
| `Fma(a, b, c)` | `fma(a, b, c)` |
| `Min(a, b)` | `min(a, b)` |
| `Max(a, b)` | `max(a, b)` |
| `Clamp(x, lo, hi)` | `clamp(x, lo, hi)` |
| `Select(c, t, f)` | `select(f, t, c)` |

#### Parallel Reduce

Tree reduction within threadgroups, then partial sums reduced on CPU or via a second kernel.

```metal
kernel void llvm2_reduce_<node_id>(
    const device float* input       [[buffer(0)]],
    device float* partial_results   [[buffer(1)]],
    threadgroup float* shared       [[threadgroup(0)]],
    uint tid  [[thread_position_in_grid]],
    uint lid  [[thread_position_in_threadgroup]],
    uint tgid [[threadgroup_position_in_grid]],
    uint tg_size [[threads_per_threadgroup]])
{
    // Load into threadgroup memory
    shared[lid] = (tid < N) ? input[tid] : <identity>;

    threadgroup_barrier(mem_flags::mem_threadgroup);

    // Tree reduction
    for (uint stride = tg_size / 2; stride > 0; stride >>= 1) {
        if (lid < stride) {
            shared[lid] = <op>(shared[lid], shared[lid + stride]);
        }
        threadgroup_barrier(mem_flags::mem_threadgroup);
    }

    // First thread writes partial result
    if (lid == 0) {
        partial_results[tgid] = shared[0];
    }
}
```

For SIMD-accelerated reduction (preferred on Apple GPUs):

```metal
kernel void llvm2_reduce_simd_<node_id>(
    const device float* input       [[buffer(0)]],
    device float* partial_results   [[buffer(1)]],
    threadgroup float* shared       [[threadgroup(0)]],
    uint tid  [[thread_position_in_grid]],
    uint lid  [[thread_position_in_threadgroup]],
    uint sgid [[simdgroup_index_in_threadgroup]],
    uint lane [[thread_index_in_simdgroup]],
    uint tgid [[threadgroup_position_in_grid]])
{
    float val = (tid < N) ? input[tid] : <identity>;

    // SIMD group reduce (hardware-accelerated, 32-wide)
    float simd_result = simd_sum(val);

    // Collect SIMD group results into threadgroup memory
    if (lane == 0) {
        shared[sgid] = simd_result;
    }
    threadgroup_barrier(mem_flags::mem_threadgroup);

    // First SIMD group reduces the partial results
    if (sgid == 0) {
        float final_val = (lid < (tg_size / 32)) ? shared[lid] : <identity>;
        final_val = simd_sum(final_val);
        if (lane == 0) {
            partial_results[tgid] = final_val;
        }
    }
}
```

The reduction operator maps from `GpuReduceOp`:

| GpuReduceOp | MSL SIMD intrinsic | MSL binary op |
|-------------|-------------------|--------------|
| `Add` | `simd_sum(val)` | `a + b` |
| `Mul` | `simd_product(val)` | `a * b` |
| `Min` | `simd_min(val)` | `min(a, b)` |
| `Max` | `simd_max(val)` | `max(a, b)` |
| `BitwiseAnd` | `simd_and(val)` | `a & b` |
| `BitwiseOr` | `simd_or(val)` | `a \| b` |
| `BitwiseXor` | `simd_xor(val)` | `a ^ b` |

#### Map-Reduce (Fused)

Fused map-reduce avoids materializing the intermediate array:

```metal
kernel void llvm2_map_reduce_<node_id>(
    const device float* a           [[buffer(0)]],
    const device float* b           [[buffer(1)]],
    device float* partial_results   [[buffer(2)]],
    threadgroup float* shared       [[threadgroup(0)]],
    uint tid  [[thread_position_in_grid]],
    uint lid  [[thread_position_in_threadgroup]],
    uint tgid [[threadgroup_position_in_grid]])
{
    // Map phase: apply per-element function
    float mapped = (tid < N) ? <map_f(a[tid], b[tid])> : <reduce_identity>;

    // Reduce phase: SIMD group reduction
    float simd_result = simd_sum(mapped);
    // ... (same threadgroup reduction as above)
}
```

#### Matrix Multiply (simdgroup_matrix)

For tMIR subgraphs identified as matrix multiplication, emit hardware-accelerated 8x8 tiled matmul:

```metal
#include <metal_simdgroup_matrix>

kernel void llvm2_matmul_<node_id>(
    const device float* A  [[buffer(0)]],
    const device float* B  [[buffer(1)]],
    device float* C        [[buffer(2)]],
    uint2 gid [[thread_position_in_grid]],
    uint2 tid_in_tg [[thread_position_in_threadgroup]])
{
    // Tile dimensions: 8x8 per simdgroup_matrix
    const uint M = <m_dim>;
    const uint K = <k_dim>;
    const uint N = <n_dim>;

    // Determine which 8x8 tile this simdgroup computes
    uint row = gid.y;  // row of output tile
    uint col = gid.x;  // col of output tile

    simdgroup_matrix<float, 8, 8> acc;
    acc = make_filled_simdgroup_matrix<float, 8, 8>(0.0f);

    // Accumulate over K dimension in 8-wide chunks
    for (uint k = 0; k < K; k += 8) {
        simdgroup_matrix<float, 8, 8> a_tile, b_tile;
        simdgroup_load(a_tile, A + row * K + k, K);
        simdgroup_load(b_tile, B + k * N + col, N);
        simdgroup_multiply_accumulate(acc, a_tile, b_tile, acc);
    }

    simdgroup_store(acc, C + row * N + col, N);
}
```

**Ref:** Apple. "Performing Calculations with simdgroup Matrices." Metal Programming Guide, 2024.

## Thread/Threadgroup/SIMD Mapping from Dispatch Plan

The dispatch plan (`DispatchPlan` from `dispatch.rs`) specifies `KernelLaunch` operations with a target of `ComputeTarget::Gpu`. Each such launch must be translated into Metal dispatch parameters.

### Grid Size Calculation

```rust
/// Compute Metal dispatch parameters from a dispatch plan kernel launch.
pub struct MetalDispatchParams {
    /// Total threads in the grid (1D, 2D, or 3D).
    pub grid_size: MTLSize,
    /// Threads per threadgroup.
    pub threadgroup_size: MTLSize,
}

/// MTLSize equivalent for the codegen layer.
pub struct MTLSize {
    pub width: u64,
    pub height: u64,
    pub depth: u64,
}

impl MetalDispatchParams {
    /// Compute dispatch params for a 1D parallel map/reduce.
    pub fn for_1d(element_count: u64, threadgroup_size: u32) -> Self {
        let tg = threadgroup_size as u64;
        // Round grid up to multiple of threadgroup size.
        let grid_width = ((element_count + tg - 1) / tg) * tg;
        MetalDispatchParams {
            grid_size: MTLSize { width: grid_width, height: 1, depth: 1 },
            threadgroup_size: MTLSize { width: tg, height: 1, depth: 1 },
        }
    }

    /// Compute dispatch params for 2D matrix operations.
    pub fn for_2d(rows: u64, cols: u64, tile_size: u32) -> Self {
        let ts = tile_size as u64;
        MetalDispatchParams {
            grid_size: MTLSize {
                width: ((cols + ts - 1) / ts) * ts,
                height: ((rows + ts - 1) / ts) * ts,
                depth: 1,
            },
            threadgroup_size: MTLSize { width: ts, height: ts, depth: 1 },
        }
    }
}
```

### Threadgroup Size Selection

The optimal threadgroup size depends on the kernel pattern and hardware:

| Pattern | Recommended Threadgroup Size | Reason |
|---------|---------------------------|--------|
| Parallel Map (simple) | 256 | Good occupancy, minimal register pressure |
| Parallel Map (complex) | 128 | Complex kernels use more registers |
| Parallel Reduce | 256 | Maximizes tree reduction parallelism |
| Map-Reduce | 256 | Same as reduce |
| MatMul (8x8 tile) | 32 (1 SIMD group) | One simdgroup per 8x8 tile |
| MatMul (larger tiles) | 128 (4 SIMD groups) | Multiple simdgroups cooperate on tile |

Metal requires `threadgroup_size <= 1024` and `threadgroup_size` must be a multiple of `simd_width` (32 on Apple GPUs). These constraints are enforced by `GpuKernelShape::is_valid()` in `gpu_semantics.rs`.

## Data Transfer: Shared Memory Buffers via MTLBuffer

### Apple UMA Model

On Apple Silicon, CPU and GPU share the same physical DRAM. The semantic encoding already models this as `encode_uma_transfer()` (identity function). The codegen materializes this as:

```rust
/// Generate host code for creating a shared Metal buffer.
///
/// On Apple UMA, `storageModeShared` avoids any data copy.
/// The CPU writes directly into the buffer; the GPU reads from
/// the same memory address.
fn emit_buffer_creation(
    buf_name: &str,
    size_bytes: u64,
    storage_mode: MTLStorageMode,
) -> String {
    format!(
        "id<MTLBuffer> {} = [device newBufferWithLength:{} options:{}];",
        buf_name,
        size_bytes,
        match storage_mode {
            MTLStorageMode::Shared => "MTLResourceStorageModeShared",
            MTLStorageMode::Private => "MTLResourceStorageModePrivate",
            MTLStorageMode::Memoryless => "MTLResourceStorageModeMemoryless",
        }
    )
}

pub enum MTLStorageMode {
    /// CPU + GPU access, no copy (UMA). Default for LLVM2.
    Shared,
    /// GPU-only access. Used for intermediate GPU-to-GPU buffers.
    Private,
    /// Tile memory (render only). Not used for compute.
    Memoryless,
}
```

### Buffer Binding Strategy

Each kernel's buffer arguments map to the dispatch plan's data flow:

```
DataTransfer { src: CpuScalar, dst: Gpu, edge_from: node_0, edge_to: node_1 }
  -> MTLBuffer with storageModeShared, bound at buffer(0)

KernelLaunch { target: Gpu, node_id: node_1 }
  -> Input buffer at buffer(0), output buffer at buffer(1)

DataTransfer { src: Gpu, dst: CpuScalar, edge_from: node_1, edge_to: node_2 }
  -> Same MTLBuffer (UMA: no copy needed, just wait for GPU completion)
```

Buffer indices are assigned sequentially per kernel:
- `buffer(0)` through `buffer(N-1)` for input buffers
- `buffer(N)` through `buffer(N+M-1)` for output buffers
- `threadgroup(0)` for threadgroup shared memory (reduce kernels)

## Runtime Integration: Host Dispatch Code

### Metal Command Submission Pipeline

LLVM2 emits host-side AArch64 code that drives the Metal runtime. The command submission follows Apple's standard pattern:

```
1. Get MTLDevice (system default GPU)
2. Create MTLCommandQueue (once per program, reused)
3. Load MTLLibrary from compiled .metallib
4. Create MTLComputePipelineState for each kernel function
5. Per dispatch:
   a. Create MTLCommandBuffer from queue
   b. Create MTLComputeCommandEncoder from buffer
   c. Set pipeline state (kernel function)
   d. Set buffer arguments
   e. Dispatch threads
   f. End encoding
   g. Commit command buffer
   h. Wait for completion (if results needed immediately)
```

### Host Code Emission

For each `KernelLaunch` targeting GPU in the dispatch plan, LLVM2 emits a sequence of Metal API calls. These are emitted as Objective-C runtime calls from AArch64 machine code:

```rust
/// Represents the host-side Metal dispatch sequence for a single kernel.
pub struct MetalHostDispatch {
    /// Pipeline state object (created once, cached).
    pub pipeline_name: String,
    /// Buffer bindings: (buffer_index, source_data_ptr, size_bytes).
    pub buffer_bindings: Vec<(u32, String, u64)>,
    /// Dispatch parameters.
    pub dispatch_params: MetalDispatchParams,
    /// Whether threadgroup memory is needed and its size.
    pub threadgroup_memory: Option<u64>,
    /// Whether to wait for completion before proceeding.
    pub wait_for_completion: bool,
}
```

The host dispatch is emitted as a function in the compiled binary. The function signature matches the tMIR computation's interface:

```
fn llvm2_dispatch_gpu_node_<id>(
    input_ptrs: &[*const u8],  // Pointers to input data
    output_ptrs: &[*mut u8],   // Pointers to output buffers
    sizes: &[u64],             // Buffer sizes
) -> Result<(), MetalError>
```

### Initialization and Caching

Metal pipeline state creation is expensive (~100us-1ms). LLVM2 generates a lazy initialization pattern:

```rust
/// Host-side initialization for Metal compute.
///
/// Generated once per program. Creates the device, queue, library,
/// and pipeline states. Pipeline states are cached for reuse.
pub struct MetalContext {
    /// System default GPU device.
    device: MTLDevice,
    /// Command queue (reusable across dispatches).
    queue: MTLCommandQueue,
    /// Compiled Metal library containing all kernels.
    library: MTLLibrary,
    /// Pipeline state cache: kernel_name -> pipeline_state.
    pipelines: HashMap<String, MTLComputePipelineState>,
}
```

This context is initialized lazily on first GPU dispatch and persists for the lifetime of the process. The compiled `.metallib` is embedded in the output Mach-O as a `__DATA,__metallib` section or loaded from a companion file.

## Metal Compiler SDK Requirements

### Build-Time Dependencies

| Tool | Purpose | Location |
|------|---------|----------|
| `xcrun -sdk macosx metal` | MSL source to AIR bitcode compiler | Xcode Command Line Tools |
| `xcrun -sdk macosx metallib` | AIR bitcode to `.metallib` archiver | Xcode Command Line Tools |

These tools are part of the Xcode Command Line Tools and are required on the build machine. They are NOT required at runtime -- the compiled `.metallib` is self-contained.

### Compile-Time Invocation

```bash
# Step 1: Compile MSL to AIR
xcrun -sdk macosx metal -c llvm2_kernels.metal -o llvm2_kernels.air

# Step 2: Archive AIR to metallib
xcrun -sdk macosx metallib llvm2_kernels.air -o llvm2_kernels.metallib
```

LLVM2's build pipeline invokes these tools as an external subprocess during compilation, similar to how it invokes the system assembler or linker.

### Runtime Dependencies

| Dependency | Purpose | Availability |
|------------|---------|-------------|
| Metal framework | GPU access | macOS 10.11+, iOS 8+ (all Apple Silicon) |
| `MTLCreateSystemDefaultDevice()` | Get GPU device | Always available on Apple Silicon |

No additional runtime dependencies. The Metal framework is part of the OS.

## Metallib Format

The `.metallib` file is an archive of compiled AIR functions. Its structure:

```
metallib file
├── Header (magic: 0x4D544C42 "MTLB", version, entry count)
├── Table of Contents (function names, offsets, sizes)
├── Function entries
│   ├── Function 0: llvm2_map_node_1
│   │   ├── AIR bitcode
│   │   └── Metadata (thread group size, buffer layout)
│   ├── Function 1: llvm2_reduce_node_3
│   │   └── ...
│   └── ...
└── String table
```

The metallib format is opaque to LLVM2 -- it is produced by Apple's `metallib` tool and consumed by the Metal framework's `MTLLibrary` loader. LLVM2 does not need to parse or generate metallib directly (in Phase 1).

## Integration with Existing LLVM2 Architecture

### Codegen Pipeline Extension

The Metal emission extends the existing `llvm2-codegen` pipeline:

```
llvm2-lower
  |
  +--> DispatchPlan (from dispatch.rs)
  |       |
  |       +--> GPU kernel launches extracted
  |
  v
llvm2-codegen
  |
  +--> AArch64 encoding (existing, for CPU code)
  |
  +--> Metal emission (NEW)
  |     |
  |     +--> MSL source generation (per GPU kernel)
  |     +--> External: xcrun metal -> .air -> .metallib
  |     +--> Host dispatch code generation (AArch64 Metal API calls)
  |
  +--> Mach-O writer (existing, extended)
        |
        +--> __TEXT,__text (CPU code including Metal host dispatch)
        +--> __DATA,__metallib (embedded compiled Metal library)
```

### New Modules

| Module | Location | Purpose |
|--------|----------|---------|
| `metal_emit.rs` | `llvm2-codegen/src/` | MSL source code generator |
| `metal_host.rs` | `llvm2-codegen/src/` | Host dispatch AArch64 code generator |
| `metal_compile.rs` | `llvm2-codegen/src/` | External Metal compiler invocation |

### Mach-O Section for Metallib

The compiled `.metallib` is embedded in the output binary as a Mach-O section. The Mach-O writer (`llvm2-codegen/src/macho/`) is extended to support a `__DATA,__metallib` section containing the raw metallib bytes.

At runtime, the Metal initialization code reads this section via `dlsym` / section inspection to load the `MTLLibrary` without filesystem access.

## Verification Chain: End-to-End Correctness

The verification chain from tMIR to GPU execution:

```
1. tMIR semantics           -- encoded by tmir_semantics.rs
2. GPU kernel semantics     -- encoded by gpu_semantics.rs
3. Equivalence proof        -- z4: tMIR_semantics == GPU_semantics (UNSAT)
4. MSL emission             -- metal_emit.rs generates MSL from proven kernel
5. Metal compilation        -- xcrun metal (trusted: Apple's compiler)
6. GPU execution            -- Metal framework (trusted: hardware)
```

Steps 1-3 are formally verified. Steps 4-6 rely on trusted components (Apple's Metal compiler and hardware). The trust boundary is at MSL emission: if the emitted MSL source correctly implements the proven semantic specification, then the GPU execution is correct.

### MSL Emission Correctness Argument

The MSL emitter generates kernel source from the same `GpuKernelPattern` enum that the semantic encoding uses. The mapping is direct:

| Semantic Encoding | MSL Output |
|-------------------|------------|
| `encode_parallel_map(input, f, n)` | `output[tid] = f(input[tid])` with bounds check |
| `encode_parallel_reduce(input, op, identity, n)` | Tree reduction with `op` and `identity` |
| `encode_map_reduce(a, b, map_f, reduce_op, id, n)` | Fused map-reduce kernel |
| `encode_uma_transfer(mem)` | `MTLBuffer` with `storageModeShared` (no copy) |

The structural correspondence between the SMT encoding and the MSL template is small enough to audit manually. A future enhancement could use the verified MSL as a test oracle (compile, run, compare against CPU output).

## Implementation Plan

### Step 1: MSL Emitter (2-3 sessions)
- Implement `MetalKernelEmitter` that takes a `GpuKernelPattern` and emits MSL source text
- Support: ParallelMap, ParallelReduce (threadgroup + SIMD variants), MapReduce, MatMul
- Expression generator for tMIR ops to MSL inline expressions
- Unit tests: verify emitted MSL compiles via `xcrun metal` (integration test)

### Step 2: Metal Compiler Integration (1 session)
- Implement `MetalCompiler` that shells out to `xcrun metal` and `xcrun metallib`
- Handle compilation errors, propagate diagnostics
- Cache compiled `.metallib` files for incremental builds

### Step 3: Host Dispatch Code Generation (2-3 sessions)
- Implement `MetalHostEmitter` that generates AArch64 code for Metal API calls
- Buffer creation (storageModeShared)
- Pipeline state creation and caching
- Command buffer / encoder / dispatch / commit / wait sequence
- Lazy initialization pattern

### Step 4: Mach-O Integration (1 session)
- Extend Mach-O writer to embed `.metallib` in `__DATA,__metallib` section
- Runtime loader to extract metallib from executable

### Step 5: End-to-End Pipeline (2 sessions)
- Wire dispatch plan GPU launches through Metal emission
- Integration test: tMIR -> dispatch plan -> Metal kernel + host code -> Mach-O
- Verify output executes correctly on real GPU

### Step 6: Phase 2 - AIR Direct Emission (future)
- Study AIR bitcode format (LLVM IR dialect)
- Emit AIR directly, bypassing MSL source generation
- Verify AIR output matches MSL compilation (regression test)

**Total estimated effort: 8-10 techlead sessions.**

## Open Questions

1. **AIR stability:** AIR is not a documented stable format. Apple may change it between Xcode versions. The MSL source path is more stable. Should LLVM2 ever target AIR directly, or always go through MSL?

2. **Async dispatch:** The current design waits for GPU completion (`waitUntilCompleted`) at each synchronization point. For sequences of GPU kernels, Metal supports chaining commands in a single command buffer without CPU round-trips. The dispatch plan could detect consecutive GPU operations and batch them.

3. **Double-buffering:** For streaming workloads, the CPU prepares the next input while the GPU processes the current one. The dispatch plan could model this with overlapping command buffers.

4. **Error handling:** Metal API calls can fail (out of memory, unsupported device, etc.). The host dispatch code needs a consistent error propagation strategy that matches LLVM2's existing error model.

5. **Cross-platform:** This design is Apple-specific. For future targets (Vulkan/SPIR-V on Linux, DirectX/DXIL on Windows), the same dispatch plan structure applies but the emission backend changes. The `MetalKernelEmitter` should implement a `GpuEmitter` trait for extensibility.
