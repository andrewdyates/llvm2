## Problem

The ANE semantic encoding (`ane_semantics.rs`) proves that a computation is correct at FP16 precision on the Neural Engine. The dispatch plan generator (`dispatch.rs`) decides when to dispatch to `ComputeTarget::NeuralEngine`. What remains is the **lowering** step: translating verified tMIR computations into CoreML models that execute on the ANE.

The ANE is not directly programmable -- it is accessed exclusively through Apple's CoreML framework. LLVM2 must generate CoreML models (`.mlmodel` / `.mlpackage`) as an intermediate representation, then invoke CoreML's compilation and execution APIs from host code.

This document specifies the complete pipeline: tMIR to CoreML model format, model compilation, data marshalling, precision handling, and runtime execution.

## CoreML Model Format

### mlmodel vs mlpackage

Apple provides two model container formats:

| Format | Extension | Contents | When to Use |
|--------|-----------|----------|-------------|
| **mlmodel** | `.mlmodel` | Protobuf-serialized model spec | Single-file, simpler, older format |
| **mlpackage** | `.mlpackage/` | Directory with model spec + weights | Large models, weight separation |

For LLVM2, `.mlmodel` is sufficient. The models generated by the compiler are computation graphs (not pre-trained neural networks), so they contain operations but no large weight blobs. The `.mlmodel` format is a single Protocol Buffers file conforming to the Core ML Model Specification.

**Ref:** Apple. "Core ML Model Specification." [coremltools documentation](https://apple.github.io/coremltools/mlmodel/index.html).

### Core ML Model Specification (Protobuf)

A CoreML model is defined by the `Model` protobuf message:

```protobuf
message Model {
    int32 specificationVersion = 1;  // Currently 8 (iOS 18 / macOS 15)
    ModelDescription description = 2;
    // One of:
    oneof Type {
        Pipeline pipeline = 200;
        MILProgram milProgram = 401;  // Modern: MIL-based programs
        // Legacy types: neuralNetwork, etc.
    }
}

message ModelDescription {
    repeated FeatureDescription input = 1;
    repeated FeatureDescription output = 10;
    string predictedFeatureName = 11;
}

message FeatureDescription {
    string name = 1;
    FeatureType type = 2;
}
```

### Model Intermediate Language (MIL)

Modern CoreML models (spec version >= 7) use MIL (Model Intermediate Language), which is an SSA-based tensor operation graph. MIL is the target for LLVM2's ANE lowering.

MIL has two dialects:
- **MIL** (high-level): operations like `matmul`, `conv`, `relu` -- corresponds directly to tMIR tensor ops
- **Backend MIL**: lower-level, target-specific. The CoreML compiler lowers MIL to Backend MIL internally.

LLVM2 targets the high-level MIL dialect. The CoreML compiler handles the rest.

```
MIL operation types relevant to LLVM2:

  Linear algebra:
    mil.matmul(x, y, transpose_x, transpose_y) -> tensor
    mil.linear(x, weight, bias) -> tensor
    mil.conv(x, weight, bias, strides, pad_type, dilations, groups) -> tensor

  Element-wise:
    mil.add(x, y) -> tensor
    mil.sub(x, y) -> tensor
    mil.mul(x, y) -> tensor
    mil.real_div(x, y) -> tensor

  Activation:
    mil.relu(x) -> tensor
    mil.leaky_relu(x, alpha) -> tensor
    mil.sigmoid(x) -> tensor
    mil.tanh(x) -> tensor
    mil.gelu(x, mode) -> tensor

  Reduction:
    mil.reduce_sum(x, axes, keep_dims) -> tensor
    mil.reduce_mean(x, axes, keep_dims) -> tensor
    mil.reduce_max(x, axes, keep_dims) -> tensor
    mil.reduce_min(x, axes, keep_dims) -> tensor

  Pooling:
    mil.avg_pool(x, kernel_sizes, strides, pad_type) -> tensor
    mil.max_pool(x, kernel_sizes, strides, pad_type) -> tensor

  Reshape:
    mil.reshape(x, shape) -> tensor
    mil.transpose(x, perm) -> tensor
```

**Ref:** Apple. "MIL Ops." [coremltools source](https://github.com/apple/coremltools/tree/main/coremltools/converters/mil/mil/ops).

## Mapping tMIR Tensor Operations to CoreML Operations

### tMIR to MIL Operation Mapping

The dispatch plan identifies compute nodes assigned to `ComputeTarget::NeuralEngine`. Each such node contains tMIR instructions that must be translated to MIL operations.

| tMIR Pattern | ANE Semantic Encoding | CoreML MIL Operation | Notes |
|-------------|----------------------|---------------------|-------|
| Matrix multiply `C[i][j] += A[i][k] * B[k][j]` | `encode_ane_gemm()` | `mil.matmul(A, B)` | Dims must be static |
| Conv2D sliding window | `encode_ane_conv2d()` | `mil.conv(input, weight, ...)` | Stride, padding from params |
| Element-wise add | `encode_ane_elementwise(Add)` | `mil.add(a, b)` | Broadcast supported |
| Element-wise sub | `encode_ane_elementwise(Sub)` | `mil.sub(a, b)` | |
| Element-wise mul | `encode_ane_elementwise(Mul)` | `mil.mul(a, b)` | |
| Element-wise div | `encode_ane_elementwise(Div)` | `mil.real_div(a, b)` | |
| ReLU activation | `encode_ane_activation(ReLU)` | `mil.relu(x)` | Exact, no precision loss |
| LeakyReLU | `encode_ane_activation(LeakyReLU)` | `mil.leaky_relu(x, alpha)` | Exact |
| Sigmoid | `encode_ane_activation(Sigmoid)` | `mil.sigmoid(x)` | FP16 approximation |
| Tanh | `encode_ane_activation(Tanh)` | `mil.tanh(x)` | FP16 approximation |
| GELU | `encode_ane_activation(GELU)` | `mil.gelu(x, mode="exact")` | FP16 approximation |
| Reduce sum | N/A (future) | `mil.reduce_sum(x, axes)` | Along specified axes |
| Reduce max | N/A (future) | `mil.reduce_max(x, axes)` | Along specified axes |
| Max pooling | N/A (future) | `mil.max_pool(x, ...)` | 2D only |
| Average pooling | N/A (future) | `mil.avg_pool(x, ...)` | 2D only |

### Fused Operation Patterns

The ANE achieves peak performance through operator fusion (documented in `ane-semantics.md`). LLVM2 should emit fused MIL patterns where possible, as the CoreML compiler recognizes these and maps them to single ANE passes:

| Fused Pattern | MIL Emission | ANE Behavior |
|--------------|-------------|-------------|
| GEMM + bias + ReLU | `mil.relu(mil.add(mil.matmul(x, w), bias))` | Single ANE pass |
| Conv + BatchNorm + ReLU | `mil.relu(mil.batch_norm(mil.conv(x, w, ...), ...))` | Single ANE pass |
| Conv + residual add + ReLU | `mil.relu(mil.add(mil.conv(x, w, ...), residual))` | Single ANE pass |
| MatMul + GELU | `mil.gelu(mil.matmul(x, w))` | Single ANE pass |

The MIL emitter should detect these patterns in the tMIR subgraph and emit the fused sequence rather than individual operations. The CoreML compiler then recognizes the pattern and fuses it for ANE execution.

## ANE Dispatch via CoreML Prediction API

### CoreML Execution Model

CoreML provides a high-level prediction API. From LLVM2's perspective, an ANE dispatch is:

```
1. Create MLModel from compiled .mlmodelc
2. Create MLMultiArray inputs (pointing to existing memory)
3. Call prediction(from: input) -> MLFeatureProvider
4. Extract MLMultiArray outputs
5. Read output data
```

The key insight: on Apple UMA, the input `MLMultiArray` can be backed by the same memory as the CPU data (zero-copy). The ANE reads from this shared memory, computes, and writes results to another `MLMultiArray` backed by shared memory.

### Host Dispatch Code Structure

For each `KernelLaunch { target: NeuralEngine, node_id }` in the dispatch plan, LLVM2 emits host code that:

```rust
/// Represents the host-side CoreML dispatch for a single ANE operation.
pub struct CoreMLHostDispatch {
    /// Path to the compiled model (.mlmodelc directory).
    pub model_path: String,
    /// Input feature descriptions: (name, shape, element_type).
    pub inputs: Vec<CoreMLFeature>,
    /// Output feature descriptions.
    pub outputs: Vec<CoreMLFeature>,
    /// Whether to use the ANE compute unit exclusively.
    pub compute_units: MLComputeUnits,
    /// Whether to wait for completion synchronously.
    pub synchronous: bool,
}

pub struct CoreMLFeature {
    /// Feature name (matches model spec).
    pub name: String,
    /// Tensor shape (static, known at compile time).
    pub shape: Vec<u64>,
    /// Element type.
    pub element_type: MLMultiArrayDataType,
}

pub enum MLComputeUnits {
    /// Let CoreML decide (default). May use CPU, GPU, or ANE.
    All,
    /// Prefer CPU + Neural Engine (skip GPU).
    CpuAndNeuralEngine,
    /// CPU only (fallback).
    CpuOnly,
}

pub enum MLMultiArrayDataType {
    Float16,   // Primary for ANE
    Float32,   // Requires quantization
    Int32,     // Limited ANE support
}
```

### Synchronous vs Asynchronous Execution

CoreML prediction can be synchronous or asynchronous:

| Mode | API | Use Case |
|------|-----|----------|
| Synchronous | `model.prediction(from: input)` | Simple, blocks until done |
| Asynchronous | `model.prediction(from: input, completionHandler: ...)` | Non-blocking, callback |

The dispatch plan's `Synchronize { target: NeuralEngine }` operation determines which mode to use. If the next consumer of ANE output is on a different target (e.g., CPU), synchronous mode ensures results are available. For chains of ANE operations, async mode with CoreML's built-in pipelining is more efficient.

## Model Compilation: mlmodel to mlmodelc

### Compilation Pipeline

```
tMIR subgraph (verified)
    |
    v
LLVM2 MIL Emitter
    |
    v
.mlmodel (protobuf, MIL program)
    |
    v
CoreML Compiler (compileModel)
    |
    v
.mlmodelc (compiled model directory)
    |
    v
MLModel.init (runtime loading)
    |
    v
ANE execution
```

### Compile-Time vs Runtime Compilation

| Strategy | When Compiled | Latency | Portability |
|----------|-------------|---------|-------------|
| **AOT** | During LLVM2 compilation | 0 runtime overhead | Compiled for build machine only |
| **JIT** | First invocation at runtime | 100ms - 5s first use | Works on any compatible device |
| **Hybrid** | Compiled model bundled, recompiled if device differs | Small first-use overhead | Best portability |

LLVM2 uses the **AOT** strategy by default: the CoreML model is compiled during LLVM2's compilation pass using `MLModel.compileModel(at:)` (Objective-C) or `coremltools.models.MLModel` (Python). The compiled `.mlmodelc` directory is embedded in the output binary.


### Embedding Compiled Models in Mach-O

The compiled `.mlmodelc` is a directory containing multiple files. Two embedding strategies:

**Strategy A: Archive and Embed**

Archive `.mlmodelc` as a tar/zip, embed in `__DATA,__coreml_model` section. At runtime, extract to a temporary directory and load.

**Strategy B: Resource Bundle**

Embed as a macOS/iOS resource bundle alongside the executable. More natural for Apple platforms but requires a bundle structure.

**Strategy C: Filesystem Companion (simplest)**

Place `.mlmodelc` alongside the output binary. Load at runtime via relative path. Simplest to implement, most debuggable.

LLVM2 uses **Strategy C** for Phase 1 and **Strategy A** for Phase 2 (single-binary deployment).

## Data Transfer: MLMultiArray and CVPixelBuffer

### MLMultiArray (Primary)

`MLMultiArray` is CoreML's tensor type. On Apple UMA, it can wrap existing memory without copying:

```objc
// Zero-copy MLMultiArray from existing buffer
NSData *data = [NSData dataWithBytesNoCopy:buffer
                                    length:size
                              freeWhenDone:NO];
MLMultiArray *array = [[MLMultiArray alloc]
    initWithDataPointer:buffer
                  shape:@[@(batch), @(channels), @(height), @(width)]
               dataType:MLMultiArrayDataTypeFloat16
                strides:@[@(C*H*W), @(H*W), @(W), @1]
            deallocator:nil
                  error:&error];
```

This is critical for performance: no memcpy between CPU and ANE domains. The dispatch plan's `DataTransfer` from CPU to ANE is implemented as MLMultiArray construction wrapping existing CPU memory.

### Data Layout

ANE prefers NCHW (batch, channels, height, width) layout, matching the `AneTensorShape` from `ane_semantics.rs`. The tMIR adapter must ensure tensors are in this layout before ANE dispatch. If the source data is in a different layout (e.g., NHWC), a transpose is needed.

| Source Layout | ANE Layout | Action |
|--------------|-----------|--------|
| NCHW | NCHW | Zero-copy |
| NHWC | NCHW | Transpose (CPU or include `mil.transpose` in model) |
| Row-major matrix | NCHW (1, 1, M, N) | Reshape (zero-copy, shape metadata only) |
| Flat array | NCHW (1, 1, 1, N) | Reshape |

### CVPixelBuffer (Image Data)

For image-processing workloads, CoreML can accept `CVPixelBuffer` inputs directly. This is relevant if tMIR processes image data (e.g., from a camera pipeline). Not a primary concern for the compiler backend but noted for completeness.

## Precision: FP16 Requirement and Quantization Strategy

### The FP16 Constraint

The ANE operates exclusively at FP16 (and INT8) precision. If the source tMIR computation uses FP32 or FP64, LLVM2 must:

1. **Quantize inputs:** Convert FP32 data to FP16 before ANE dispatch
2. **Dequantize outputs:** Convert FP16 results back to FP32 after ANE dispatch
3. **Verify bounded error:** Prove the quantization error is acceptable (using `encode_bounded_error_check()` from `ane_semantics.rs`)

### Quantization Pipeline

```
FP32 tMIR computation (verified)
    |
    v
Bounded error proof: |FP32_result - FP16_result| < epsilon
    |  (via ane_semantics.rs::encode_bounded_error_check)
    v
If proof succeeds: emit FP16 CoreML model
    |
    v
Host code:
  1. FP32 input -> FP16 conversion (CPU, vectorized via NEON)
  2. FP16 input -> ANE via CoreML prediction
  3. FP16 output -> FP32 conversion (CPU)
```

### FP16 Conversion Implementation

```rust

```

The conversion cost is accounted for in the dispatch plan's transfer cost estimate. For Apple UMA, the "transfer" from CPU to ANE is not a memory copy but a precision conversion.

### Quantization Error Bounds

From `ane_semantics.rs`, the error bounds are:

| Operation | Max Relative Error (FP16) | Absolute Error Formula |
|-----------|--------------------------|----------------------|
| Element-wise add/sub | ~0.1% | 2^{-10} * max(\|a\|, \|b\|) |
| Element-wise mul | ~0.1% | 2^{-10} * \|a\| * \|b\| |
| GEMM (M x K) | Grows with K | K * 2^{-10} * max(\|A\|) * max(\|B\|) |
| Conv2D (C_in x KH x KW) | Grows with kernel | C_in*KH*KW * 2^{-10} |
| ReLU | 0% (exact) | 0 |

The unified solver uses these bounds to decide ANE legality. If the application's error tolerance (from tMIR proof annotations) is tighter than the FP16 error bound, ANE is illegal for that subgraph.

## Runtime: MLModel Loading, Prediction, and Async Execution

### Model Lifecycle

```
Program startup
    |
    v
Lazy initialization (first ANE dispatch)
    |
    +-- Load compiled .mlmodelc -> MLModel
    |     (10-100ms, done once)
    |
    +-- Configure compute units (prefer ANE)
    |
    v
Per-dispatch:
    |
    +-- Wrap input data as MLMultiArray (zero-copy on UMA)
    |
    +-- Call prediction API
    |     |
    |     +-- CoreML routes to ANE (if operation supported)
    |     +-- Or GPU fallback (if ANE cannot handle)
    |     +-- Or CPU fallback (last resort)
    |
    +-- Extract output MLMultiArray
    |
    +-- (Optional) Convert FP16 output back to FP32
    |
    v
Results available to next dispatch plan step
```

### Host Code Structure

```rust
/// CoreML ANE execution context, initialized lazily.
pub struct CoreMLContext {
    /// Loaded and compiled MLModel instances.
    /// Key: model identifier (derived from compute node ID).
    models: HashMap<String, MLModel>,

    /// MLModelConfiguration for ANE targeting.
    config: MLModelConfiguration,
}

impl CoreMLContext {
    /// Initialize with ANE preference.
    pub fn new() -> Self {
        let mut config = MLModelConfiguration::new();
        config.compute_units = MLComputeUnits::CpuAndNeuralEngine;
        CoreMLContext {
            models: HashMap::new(),
            config,
        }
    }

    /// Load a compiled model (lazy, cached).
    pub fn load_model(&mut self, model_id: &str, mlmodelc_path: &str) -> &MLModel {
        self.models.entry(model_id.to_string()).or_insert_with(|| {
            MLModel::init(contentsOf: mlmodelc_path, configuration: &self.config)
        })
    }

    /// Execute prediction.
    pub fn predict(
        &self,
        model_id: &str,
        inputs: &[(String, MLMultiArray)],
    ) -> Result<HashMap<String, MLMultiArray>, CoreMLError> {
        let model = &self.models[model_id];
        let input_provider = MLDictionaryFeatureProvider::new(inputs);
        let output = model.prediction(from: input_provider)?;
        // Extract output features
        Ok(extract_outputs(output))
    }
}
```

### Async Execution Pattern

For non-blocking ANE dispatch (used when the dispatch plan has a `Synchronize` deferred):

```rust
/// Async ANE dispatch using CoreML's prediction API.
///
/// Returns a handle that can be waited on later.
pub fn predict_async(
    context: &CoreMLContext,
    model_id: &str,
    inputs: &[(String, MLMultiArray)],
    completion: impl FnOnce(Result<HashMap<String, MLMultiArray>, CoreMLError>),
) {
    let model = &context.models[model_id];
    let input_provider = MLDictionaryFeatureProvider::new(inputs);
    model.prediction(from: input_provider, completionHandler: |result, error| {
        if let Some(error) = error {
            completion(Err(CoreMLError::from(error)));
        } else {
            completion(Ok(extract_outputs(result)));
        }
    });
}
```

The dispatch plan's `Synchronize { target: NeuralEngine, node_id }` becomes a wait on the async completion handle.

## Limitations: ANE-Supported Operations Subset

### What the ANE Cannot Do

The ANE is a fixed-function accelerator. It excels at a narrow set of tensor operations and fails at everything else. LLVM2 must enforce these limitations at the legality check stage (before code generation).

| Category | Unsupported Operations | Fallback Target |
|----------|----------------------|----------------|
| **Control flow** | Data-dependent branches, loops with variable bounds | CPU |
| **Irregular memory** | Scatter, gather with non-contiguous indices | GPU |
| **Integer arithmetic** | General-purpose integer compute (non-tensor) | CPU |
| **FP32/FP64** | Full-precision floating point (no FP16 equivalent) | CPU or GPU |
| **Dynamic shapes** | Tensors whose dimensions are not known at compile time | CPU or GPU |
| **Custom activations** | Activation functions not in the supported set | CPU (or approximate) |
| **Large reductions** | Reduction over very large tensors (>10M elements) | GPU |
| **Sparse operations** | Sparse matrix operations | CPU |

### Legality Check (from dispatch.rs)

The dispatch plan generator already handles this via `is_async_target(ComputeTarget::NeuralEngine)`. The target analysis module (`target_analysis.rs`) determines ANE legality. The CoreML emitter should assert that it only receives operations that passed the legality check:

```rust
/// Assert that a compute node is ANE-legal before emitting CoreML.
///
/// Panics if the node contains unsupported operations. This is a
/// defense-in-depth check -- the dispatch plan generator should never
/// assign an illegal node to ANE.
fn assert_ane_legal(node: &ComputeNode) {
    for instr in &node.instructions {
        assert!(
            is_ane_supported(instr),
            "Compute node {} contains ANE-illegal instruction: {:?}. \
             The dispatch plan generator has a bug.",
            node.id, instr
        );
    }
}
```

### CoreML's Internal Fallback

Even if LLVM2 correctly targets ANE, CoreML may internally fall back to GPU or CPU for specific operations. This happens transparently -- CoreML's ANE compiler decides which ops run on ANE vs GPU within a single model. LLVM2's correctness is not affected (the semantic equivalence proof covers all execution targets), but performance may be lower than expected.

To detect this, LLVM2 can query the model's compute plan after compilation:

```swift
// Check which compute units CoreML selected for each operation
let plan = try await MLComputePlan(contentsOf: modelURL,
                                    configuration: config)
for layer in plan.modelStructure.layers {
    let unit = plan.computeDeviceUsage(for: layer)
    // unit tells us: neuralEngine, cpu, or gpu
}
```

This is a diagnostic tool, not a correctness concern. If CoreML falls back to GPU for an operation LLVM2 intended for ANE, the result is correct but may be slower.

**Ref:** Apple. "MLComputePlan." Apple Developer Documentation, 2024 (iOS 17+ / macOS 14+).

CoreML's `.mlmodel` format is Protocol Buffers. The MIL emitter needs:

3. **Option C:** Generate the protobuf bytes directly (manual serialization)

Option A is preferred for a Rust-native pipeline. The CoreML model protobuf schema is available from Apple's [coremltools repository](https://github.com/apple/coremltools/tree/main/mlmodel/format).

## Verification Chain: End-to-End Correctness

```
1. tMIR semantics            -- encoded by tmir_semantics.rs
2. ANE operation semantics   -- encoded by ane_semantics.rs (FP16 precision model)
3. Equivalence proof         -- z4: tMIR_semantics == ANE_semantics (with bounded error)
4. MIL emission              -- coreml_emit.rs generates MIL from proven operations
5. CoreML compilation        -- Apple compileModel (trusted: Apple's compiler)
6. ANE execution             -- CoreML prediction (trusted: hardware)
```

Steps 1-3 are formally verified. Steps 4-6 rely on trusted components. The trust boundary is at MIL emission: if the emitted MIL program correctly implements the proven ANE semantic specification, then the ANE execution is correct (within the proven error bounds).

### MIL Emission Correctness Argument

The MIL emitter maps from the same `AneOperation` types used in the semantic encoding. The mapping is structural:

| ANE Semantic Encoding | CoreML MIL Output |
|----------------------|-------------------|
| `encode_ane_gemm(a, b, m, k, n)` | `mil.matmul(a, b)` with shape [m, k] x [k, n] |
| `encode_ane_conv2d(input, kernel, bias, params)` | `mil.conv(input, kernel, bias, strides, pad_type)` |
| `encode_ane_activation(input, ReLU, n)` | `mil.relu(input)` |
| `encode_ane_activation(input, Sigmoid, n)` | `mil.sigmoid(input)` |
| `encode_ane_elementwise(a, b, Add, n)` | `mil.add(a, b)` |
| `encode_fp16_quantize(array, n)` | Input data conversion (host code) |

## Implementation Plan

### Step 1: CoreML Model Types (1 session)
- Define Rust types for CoreML model spec (protobuf messages)
- Define MIL operation types matching `AneOperation` from `ane_semantics.rs`
- Set up `prost` protobuf generation from CoreML schema

### Step 2: MIL Program Emitter (2-3 sessions)
- Implement `CoreMLEmitter` that takes a compute subgraph and emits MIL program

- Handle tensor shape annotations and data types
- Fusion pattern detection and emission

### Step 3: mlmodel Serializer (1 session)
- Serialize MIL program to `.mlmodel` protobuf format

## Open Questions

1. **coremltools dependency:** Should LLVM2 depend on `coremltools` (Python) for model generation, or implement a pure Rust mlmodel serializer? Python dependency is simpler but adds a build requirement. Pure Rust is more self-contained.

2. **Model granularity:** Should LLVM2 emit one CoreML model per ANE-targeted compute node, or merge consecutive ANE nodes into a single model? Single-model-per-node is simpler but misses fusion opportunities. Merged models enable CoreML's internal fusion optimizer.

3. **M4 BF16 support:** Apple M4 adds BF16 support to the ANE. Should LLVM2 support BF16 as an alternative to FP16? BF16 has larger range but less precision (7 vs 10 mantissa bits). Some algorithms prefer BF16 (e.g., training).

4. **INT8 quantization:** The ANE also supports INT8. For inference workloads with INT8-tolerant algorithms, INT8 provides 2x throughput over FP16. This requires a more complex quantization pipeline (calibration, scale factors). Defer to Phase 2.

5. **CoreML version compatibility:** CoreML's MIL spec evolves across macOS/iOS versions. Should LLVM2 target a minimum spec version (e.g., 7 for MIL support) or always use the latest? Targeting a minimum version ensures broader compatibility.

6. **MLComputePlan diagnostics:** Should LLVM2 check CoreML's compute plan at compile time to verify ANE targeting, or trust the legality check and accept CoreML's internal fallback decisions?
