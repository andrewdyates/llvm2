## Overview

This document audits the z4-bindings API surface to determine what LLVM2 needs
for formal verification, what z4 already provides, and what gaps must be filled.
The goal is to replace LLVM2's current CLI-subprocess approach with native
z4 Rust API integration.

---

## Current LLVM2-z4 Integration: CLI Subprocess Bridge

**Source:** `crates/llvm2-verify/src/z4_bridge.rs`

LLVM2 currently uses two verification backends:

### Backend 1: CLI Subprocess (always available)

```
ProofObligation
  -> to_smt2() -> SMT-LIB2 string
  -> write to temp file
  -> invoke z3/z4 binary with -smt2 flag
  -> parse stdout (sat/unsat/unknown)
  -> extract counterexample from (get-value ...) output
```

**Implementation:** `verify_with_cli()` in `z4_bridge.rs`
- Searches PATH for `z3` then `z4` binary via `which`
- Writes SMT-LIB2 to `/tmp/llvm2_verify_<pid>_<counter>.smt2`
- Parses three model formats: `#xHEX`, `#bBINARY`, `(_ bvDECIMAL WIDTH)`
- Configurable timeout (default 5000ms)
- Works today with z3 -- integration tests pass

### Backend 2: z4 Native Rust API (feature-gated, partial)

```
ProofObligation
  -> translate SmtExpr tree to z4::Term tree
  -> z4::Solver.assert_term(negated_equivalence)
  -> z4::Solver.check_sat()
  -> extract model for counterexample
```

**Implementation:** `verify_with_z4_api()` in `z4_bridge.rs` (behind `#[cfg(feature = "z4")]`)
- Creates `z4::Solver` with `Logic::QfBv`
- Declares BV inputs via `solver.declare_const(name, Sort::BitVec(width))`
- Translates SmtExpr tree recursively via `translate_expr_to_z4()`
- **Supports:** All BV operations (add, sub, mul, div, neg, and/or/xor, shifts,
  comparisons, extract, concat, zero_ext, sign_ext, ite)
- **Does NOT support:** Array operations (Select, Store, ConstArray), FP operations,
  Uninterpreted Functions

**The gap:** All memory proofs use array theory (QF_ABV) and cannot run through
the native API. They fall back to CLI subprocess.

## What SMT Theories LLVM2 Needs

### QF_BV -- Quantifier-Free Bitvectors (NEEDED, AVAILABLE)

**Used for:** Arithmetic lowering (add/sub/mul/neg), comparison lowering (10 ops),
NZCV flag proofs, peephole identities, constant folding, copy propagation.

**Status:** Fully supported in both CLI and native API backends.

**Operations needed:**
- BV arithmetic: `bvadd`, `bvsub`, `bvmul`, `bvsdiv`, `bvudiv`, `bvneg`
- BV bitwise: `bvand`, `bvor`, `bvxor`, `bvshl`, `bvlshr`, `bvashr`
- BV comparison: `bvslt`, `bvsle`, `bvsgt`, `bvsge`, `bvult`, `bvule`, `bvugt`, `bvuge`
- BV manipulation: `extract`, `concat`, `zero_extend`, `sign_extend`
- Boolean: `and`, `or`, `not`, `eq`, `ite`
- Constants: `(_ bvN W)`, `true`, `false`

### QF_ABV -- Quantifier-Free Arrays + Bitvectors (NEEDED, PARTIALLY AVAILABLE)

**Used for:** Memory model verification (27 proof obligations). Memory modeled as
`Array(BitVec64, BitVec8)`.

**Operations needed:**
- `select(array, index)` -- read byte from memory
- `store(array, index, value)` -- write byte to memory
- `((as const (Array K V)) default)` -- constant array initialization

**Status in z4-bindings:**
- `Expr::select(index)` -- AVAILABLE (`~/z4/crates/z4-bindings/src/expr/arrays.rs:114`)
- `Expr::store(index, value)` -- AVAILABLE (`~/z4/crates/z4-bindings/src/expr/arrays.rs:131`)
- `Expr::const_array(index_sort, value)` -- AVAILABLE (`~/z4/crates/z4-bindings/src/expr/arrays.rs:145`)
- `Expr::try_select/try_store/try_const_array` -- AVAILABLE (fallible versions)
- `Sort::Array(ArraySort)` -- AVAILABLE, with `Sort::array(index, element)` constructor
- SMT-LIB2 serialization -- AVAILABLE via Display trait

**Status in z4 solver:**
- `z4-dpll` has `bv_array_smoke_tests.rs` -- array theory is under development
- The native `translate_expr_to_z4()` in LLVM2 currently returns:
  `Err("Array theory (QF_ABV) not yet supported in z4 native API")`

**Gap:** z4 solver's DPLL(T) framework needs a complete array theory decision
procedure. The bindings API is ready -- the solver backend is not.

### QF_FP -- Quantifier-Free Floating-Point (FUTURE, PARTIALLY AVAILABLE)

**Used for:** Floating-point instruction lowering verification (FADD, FMUL, FDIV,
FNEG, FCVT). Currently deferred in the verification architecture.

**Operations needed:**
- FP arithmetic: `fp.add`, `fp.mul`, `fp.div`, `fp.neg`
- FP comparison: `fp.eq`, `fp.lt`, `fp.leq`
- FP classification: `fp.isNaN`, `fp.isInfinite`, `fp.isZero`
- FP conversion: `fp.to_sbv`, `fp.to_ubv`, `sbv_to_fp`, `ubv_to_fp`
- FP sorts: `(_ FloatingPoint 8 24)` (F32), `(_ FloatingPoint 11 53)` (F64)

**Status in z4-bindings:**
- `Expr` FP operations -- AVAILABLE (`~/z4/crates/z4-bindings/src/expr/fp.rs`)
  - `fp_add`, `fp_mul`, `fp_div`, `fp_neg`, `fp_eq`, `fp_lt` etc.
  - FP constants: `fp_plus_infinity`, `fp_minus_infinity`, `fp_nan`, `fp_plus_zero`, `fp_minus_zero`
  - Rounding modes: RNE, RNA, RTP, RTN, RTZ
  - `fp_predict.rs` module for FP classification predicates
- `Sort::FloatingPoint(eb, sb)` -- AVAILABLE
- SMT-LIB2 serialization -- AVAILABLE

**Status in LLVM2 z4_bridge.rs:**
- SMT-LIB2 generation: `infer_logic()` detects FP and emits `QF_BVFP`
- `generate_smt2_query()` declares FP inputs with correct sort
- `rounding_mode_to_smt2()` serializes all 5 rounding modes
- Native API translation: returns `Err("Floating-point theory not yet supported")`

**Gap:** z4 solver's FP theory decision procedure. The bindings and LLVM2's
SMT-LIB2 generation are both ready.

### QF_UFBV -- Uninterpreted Functions + Bitvectors (FUTURE, AVAILABLE)

**Used for:** Abstracting complex operations (e.g., treating `mul` as
uninterpreted when exact semantics are not needed for a proof).

**Status in z4-bindings:**
- `Expr::uf(name, args, result_sort)` -- AVAILABLE
- `ExprValue::UFDecl` -- AVAILABLE
- SMT-LIB2 serialization -- AVAILABLE

**Status in LLVM2:**
- `infer_logic()` detects UF and emits `QF_UFBV`
- Native API: returns `Err("Uninterpreted function theory not yet supported")`

**Priority:** Low. UF is useful for abstraction but not required for correctness
proofs of concrete instruction lowering.

## z4-Bindings API Surface Needed by LLVM2

### Context and Program Creation

```rust
use z4_bindings::{Z4Program, Sort, Expr, Constraint};

// Create a verification program
let mut prog = Z4Program::qf_bv();           // QF_BV for arithmetic proofs
let mut prog = Z4Program::new("QF_ABV");      // QF_ABV for memory proofs
prog.produce_models();                         // enable counterexample extraction

// For BMC-style bounded verification
let mut prog = Z4Program::for_bmc();
```

**Status:** All AVAILABLE in z4-bindings.

### Variable Declaration

```rust
// Bitvector inputs
let x = prog.declare_const("x", Sort::bv32());
let addr = prog.declare_const("addr", Sort::bitvec(64));

// Array inputs (memory)
let mem = prog.declare_const("mem",
    Sort::array(Sort::bitvec(64), Sort::bitvec(8)));

// FP inputs
let f = prog.declare_const("f", Sort::fp64());
```

**Status:** All AVAILABLE.

### BV Operations (Complete)

```rust
// Arithmetic
let sum = a.bvadd(b);
let diff = a.bvsub(b);
let prod = a.bvmul(b);
let sdiv = a.bvsdiv(b);
let udiv = a.bvudiv(b);
let neg = a.bvneg();

// Bitwise
let and = a.bvand(b);
let or = a.bvor(b);
let xor = a.bvxor(b);
let shl = a.bvshl(b);
let lshr = a.bvlshr(b);
let ashr = a.bvashr(b);

// Comparison (returns Bool-sorted Expr)
let eq = a.eq(b);
let slt = a.bvslt(b);
let ult = a.bvult(b);
// ... all 8 signed/unsigned comparisons

// Manipulation
let bits = x.extract(7, 0);           // extract bits [7:0]
let wide = x.zero_extend(32);         // zero-extend by 32 bits
let swide = x.sign_extend(32);        // sign-extend by 32 bits
let cat = hi.concat(lo);              // concatenate

// Constants
let c = Expr::bitvec_const(42i32, 32);
let zero = Expr::bitvec_const(0, 64);
```

**Status:** All AVAILABLE. LLVM2's `translate_expr_to_z4()` already maps all
of these to z4 API calls.

### Array Operations (Memory Model)

```rust
// Constant array (all bytes initialized to default)
let mem = Expr::const_array(Sort::bitvec(64), Expr::bitvec_const(0, 8));

// Read byte from memory
let byte = mem.select(addr);

// Write byte to memory
let mem2 = mem.store(addr, byte_val);

// Chained store-load (roundtrip)
let mem3 = mem.store(addr, val);
let loaded = mem3.select(addr);
```

**Status:** API AVAILABLE in z4-bindings. Solver backend NOT YET AVAILABLE in z4.

### Assertion and Checking

```rust
// Assert the negated equivalence (tmir != aarch64)
prog.assert(tmir_result.eq(aarch64_result).not());

// Add preconditions
prog.assert(precondition);

// Check satisfiability
prog.check_sat();
prog.get_model();

// Serialize to SMT-LIB2
let smt2 = prog.to_string();
```

**Status:** All AVAILABLE.

## Performance: CLI Subprocess vs Native API

### CLI Subprocess Overhead

1. **Serialization:** SmtExpr tree -> SMT-LIB2 string (~0.1ms per proof)
2. **File I/O:** Write to /tmp, solver reads file (~0.1ms)
3. **Process spawn:** `fork + exec(z3)` (~5-10ms)
4. **Parse output:** String parsing of sat/unsat/model (~0.1ms)
5. **Per-proof total overhead:** ~5-15ms (dominated by process spawn)

For 27 memory proofs: ~135-405ms overhead beyond solving time.

### Native API (Expected)

1. **Translation:** SmtExpr tree -> z4::Term tree (~0.05ms per proof)
2. **Assertion:** In-process, no serialization (~0.01ms)
3. **No process spawn:** zero overhead
4. **Model extraction:** Direct API access (~0.01ms)
5. **Per-proof total overhead:** ~0.1ms

For 27 memory proofs: ~2.7ms overhead -- a **50-150x reduction**.

### Solving Time (Dominates Both)

QF_ABV proofs at the complexity of LLVM2's memory model (byte-level
select/store chains, 2-8 bytes) typically solve in:
- z3: 10-100ms per proof
- z4 (projected): similar, once array theory is complete

**Conclusion:** For development iteration (running proofs 100+ times/day), the
CLI overhead is noticeable but tolerable. For CI gates (running once per merge),
it is negligible. The native API primarily benefits developer experience.

## Feature Flag Design

### Current State

```toml
# crates/llvm2-verify/Cargo.toml
[features]
default = []
z4 = []  # Feature exists but only enables the native API code path
```

The `z4` feature gates the native API backend. When disabled, `verify_with_z4()`
falls through to `verify_with_cli()`.

### Recommended Design

```toml
[features]
default = []
z4 = ["dep:z4-bindings"]           # Native z4 API (in-process)

[dependencies]
z4-bindings = { git = "...", optional = true, default-features = false }
```

**Behavior matrix:**

| Feature | Backend | Memory proofs | Performance |
|---------|---------|---------------|-------------|
| (none) | Mock evaluation only | Evaluation-based | Fast (~1ms) |
| (none) + z3 on PATH | CLI subprocess | z3-backed | Medium (~5s for all) |
| `z4` | Native z4 API | In-process | Fast (~1s for all) |
| `z4` + z3 on PATH | Native z4 + CLI fallback | Best available | Fast with fallback |

### API Selection Logic

```rust
pub fn verify_with_z4(obligation: &ProofObligation, config: &Z4Config) -> Z4Result {
    #[cfg(feature = "z4")]
    {
        let result = verify_with_z4_api(obligation, config);
        match &result {
            Z4Result::Error(msg) if msg.contains("not yet supported") => {
                // Theory not supported in z4 native -- fall back to CLI
                verify_with_cli(obligation, config)
            }
            _ => return result,
        }
    }

    #[cfg(not(feature = "z4"))]
    {
        verify_with_cli(obligation, config)
    }
}
```

This provides graceful degradation: native API for what z4 supports, CLI
fallback for array/FP theories until z4 catches up.

## Migration Path: CLI to Native

### Step 1: Validate CLI Completeness (Ready Now)

Run all 27 memory proofs through `verify_with_cli()` with z3.
Verify each returns `Z4Result::Verified`. This establishes the baseline.

```rust
let config = Z4Config::default();
for obligation in memory_proofs::all_memory_proofs() {
    let result = verify_with_cli(&obligation, &config);
    assert_eq!(result, Z4Result::Verified);
}
```

### Step 2: Add z4-bindings Dependency (When z4 Array Theory Ships)

```toml
[dependencies.z4-bindings]
git = "https://github.com/dropbox-ai-prototypes/z4"
optional = true
features = []
```

### Step 3: Implement Array Translation in `translate_expr_to_z4()`

Replace the current error stubs with actual translations:

```rust
SmtExpr::Select { array, index } => {
    let a = translate_expr_to_z4(array, solver, var_terms)?;
    let i = translate_expr_to_z4(index, solver, var_terms)?;
    Ok(a.select(i))
}
SmtExpr::Store { array, index, value } => {
    let a = translate_expr_to_z4(array, solver, var_terms)?;
    let i = translate_expr_to_z4(index, solver, var_terms)?;
    let v = translate_expr_to_z4(value, solver, var_terms)?;
    Ok(a.store(i, v))
}
SmtExpr::ConstArray { index_sort, value } => {
    let v = translate_expr_to_z4(value, solver, var_terms)?;
    let idx_sort = translate_sort(index_sort);
    Ok(z4::Expr::const_array(idx_sort, v))
}
```

### Step 4: Update Solver Logic Selection

Change the solver creation from hardcoded `Logic::QfBv` to inferred:

```rust
let logic = match infer_logic(&obligation.negated_equivalence()) {
    "QF_BV"   => Logic::QfBv,
    "QF_ABV"  => Logic::QfAbv,
    "QF_BVFP" => Logic::QfBvfp,
    _         => Logic::All,
};
let solver = Solver::try_new(logic)?;
```

### Step 5: Run Regression Suite

All 27 memory proofs + all existing BV proofs must produce identical
results through both CLI and native API paths.

## z4 Memory Model Comparison

z4-bindings includes a `MemoryModel` struct (`memory.rs`) that is more
sophisticated than LLVM2's memory encoding:

| Feature | z4 MemoryModel | LLVM2 memory_proofs.rs |
|---------|---------------|----------------------|
| Pointer representation | Split: (object_id:32, offset:32) | Flat: BitVec(64) |
| Memory array | `Array(BV64, BV8)` | `Array(BV64, BV8)` |
| Object tracking | `object_size` and `object_valid` arrays | None |
| Allocation | `allocate()` with monotonic object IDs | None |
| Deallocation | `deallocate()` marks invalid | None |
| Bounds checking | `read_ok(ptr, size)` checks object bounds | None |
| Stack frames | Push/pop with automatic deallocation | None |
| Non-aliasing | By construction (different object_id) | By precondition (disjoint addresses) |

**Opportunity:** When LLVM2 integrates z4-bindings, it could adopt z4's
`MemoryModel` for richer verification:
- Use split-pointer representation to leverage tMIR's `ValidBorrow` annotations
- Use `object_valid` to verify use-after-free prevention
- Use `read_ok` to verify InBounds annotations

**Risk:** z4's MemoryModel is designed for general Rust verification (bounded
model checking of Rust programs), not specifically for compiler backend
verification. LLVM2's simpler flat-address model may be more appropriate for
instruction-level lowering proofs. The z4 MemoryModel would be more useful for
whole-function verification (verify.rs `verify_transformation()`).

## Recommendations

1. **Immediate (no z4 changes needed):** Run all 27 memory proofs through z3
   CLI to get formal verification today. The infrastructure is ready.

2. **Short-term (z4 array theory):** File an issue on z4 requesting array theory
   support in z4-dpll. LLVM2 is the primary consumer. The bindings are ready;
   only the solver backend is missing.

3. **Medium-term (native integration):** Once z4 has array theory, implement
   `translate_expr_to_z4()` for Select/Store/ConstArray and switch to native API.
   This eliminates subprocess overhead.

4. **Long-term (rich memory model):** Consider adopting z4's split-pointer
   MemoryModel for whole-function verification, while keeping the simple
   flat-address model for per-instruction lowering proofs.

5. **Feature flag:** Keep z4 as an optional dependency. The core compilation
   pipeline must never depend on the verifier. Verification is offline tooling.
