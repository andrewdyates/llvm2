# Pulse Metrics System

Pulse is a programmatic health monitoring system that collects project metrics, detects issues, and writes flag files for Manager review.

**Entry point:** `./ai_template_scripts/pulse.py` or `python -m ai_template_scripts.pulse`

## Architecture Overview

Pulse is organized into 16 modules (~7300 LOC total):

| Module | Responsibility |
|--------|----------------|
| `core.py` | Orchestration: `collect_metrics()`, `pulse_once()`, `pulse_watch()` |
| `test_metrics.py` | Test counts, framework detection, proof coverage |
| `issue_metrics.py` | GitHub issues, velocity, blockers, reopen metrics |
| `git_metrics.py` | Repo status, quality metrics, doc claim verification |
| `flags.py` | Threshold checking, flag file writing with summaries |
| `system.py` | Memory, disk usage, rate limits, disk bloat detection |
| `session_metrics.py` | Active AI sessions, untraceable failures |
| `dirty_tracking.py` | Read-only invariant enforcement |
| `code_metrics.py` | Lines of code, file sizes, complexity tracking |
| `process_metrics.py` | Long-running processes, crash detection |
| `config.py` | Configuration loading from pulse.toml |
| `storage.py` | Metrics persistence, rotation, compaction |
| `output.py` | Formatted terminal output |
| `constants.py` | Path constants, patterns, default thresholds |
| `__init__.py` | Public API re-exports |
| `__main__.py` | CLI entry point |

### Data Flow

```
collect_metrics()
    |
    +--> [~20 collector calls] --> metrics dict
    |
    v
check_thresholds(metrics) --> flags list
    |
    v
write_metrics(metrics) + write_flags(flags)
```

## Metrics Collected

### Code Metrics
- **loc**: Lines of code by file type (Python, Rust, Markdown, etc.)
- **large_files**: Files exceeding `max_file_lines` threshold
- **forbidden_ci**: GitHub Actions workflows (not supported in dropbox-ai-prototypes)
- **complexity**: Cyclomatic complexity with day-over-day delta tracking

### Test & Proof Metrics
- **tests.count**: Total test count
- **tests.by_framework**: Test counts by framework (pytest, cargo, jest, etc.)
- **tests.recent_24h**: Recent test runs (passed, failed, total)
- **tests.orphaned_tests**: Python test files not reachable from conftest.py
- **tests.coverage**: Test coverage percentage (if coverage data exists)
- **proofs.kani**: Kani proof coverage (harnesses, functions, coverage_pct)
- **proofs.kani.passing/failing/unexecuted/timeout/oom**: Verification status breakdown
- **proofs.tla2**: TLA+ specs
- **proofs.lean**: Lean theorems

### Kani Verification Status Tracking (#2263)

Pulse reads Kani verification status from `kani_status.json` (generated by `kani_runner.py`):

```json
{
  "harnesses": {
    "z4-sat::proof_vivify_subset": {
      "status": "passed",
      "duration_sec": 4.9,
      "verified_at": "2026-02-01T10:00:00Z",
      "commit": "abc123"
    },
    "z4-sat::proof_htr_resolution_valid": {
      "status": "timeout",
      "timeout_sec": 300,
      "last_attempt": "2026-02-02T15:00:00Z",
      "commit": "def456"
    }
  },
  "summary": {
    "passed": 180,
    "failed": 2,
    "timeout": 15,
    "oom": 3,
    "not_run": 40
  },
  "last_updated": "2026-02-03T12:00:00Z"
}
```

**Status values:**
| Status | Description |
|--------|-------------|
| `passed` | Verification succeeded |
| `failed` | Verification failed (proof doesn't hold) |
| `timeout` | Exceeded timeout limit |
| `oom` | Out of memory (exit code 137/139) |
| `not_run` | Harness exists but never executed |
| `error` | Kani/cargo invocation error |

**Usage:**
```bash
# Run all proofs with 5-minute timeout
python3 -m ai_template_scripts.kani_runner --timeout 300

# Run only not_run proofs
python3 -m ai_template_scripts.kani_runner --filter not_run

# Audit: compare tracking file to actual harnesses
python3 -m ai_template_scripts.kani_runner --audit

# Dry run: discover harnesses without executing
python3 -m ai_template_scripts.kani_runner --dry-run
```

The `kani_unverified` flag triggers when >20% of harnesses are unverified (not_run + timeout + oom).

### Issue Metrics
- **issues**: Open, in-progress, blocked, P0/P1/P2/P3 counts
- **velocity**: Issues opened/closed in last 7/30 days
- **blocked_issue_list**: List of blocked issues with titles
- **blocked_missing_reason**: Blocked issues without `Blocked:` text in body
- **stale_blockers**: Blocked issues where the blocker is already closed
- **long_blocked**: Blocked issues with no progress >7 days
- **issues_reopened**: Reopen rate tracking (indicates closure quality)

### System Metrics
- **system.memory**: Memory usage percentage
- **system.disk**: Disk usage percentage
- **system.disk_bloat**: Large files, bloated directories, artifacts
- **system.gh_rate_limit**: GitHub API rate limit status

### Git Metrics
- **git.branch**: Current branch
- **git.head**: Current HEAD commit
- **git.dirty**: Whether working tree is dirty
- **git.operation_in_progress**: Incomplete rebase/merge/cherry-pick
- **git_deps**: Outdated git dependencies in Cargo.toml

### Session Metrics
- **active_sessions**: Count of running AI sessions
- **active_session_details**: Role and PID of each session
- **long_running_processes**: Processes exceeding runtime threshold
- **crashes_24h**: Recent crash count (real vs stale_connection)
- **untraceable_failures**: Worker failures with no working_issues

### Quality Metrics
- **quality.test_code_ratio**: Lines of test code / production code
- **quality.type_coverage**: Python type annotation coverage
- **consolidation**: Multi-file modules that should be combined
- **doc_claims**: Documentation claim verification

## Flags

Flags are written to `.flags/` directory. Each flag file contains a timestamp and actionable summary.

### Resource Flags
| Flag | Trigger | Action |
|------|---------|--------|
| `memory` | Memory >=80% (warning) or >=90% (critical) | Check severity in flag file; free memory or kill processes |
| `disk` | Disk >=80% (warning) or >=90% (critical) | Check severity in flag file; clean up artifacts |
| `disk_bloat` | Large files, bloated tests/, artifacts | See flag file for details |

### Quality Flags
| Flag | Trigger | Action |
|------|---------|--------|
| `large_files` | >=3 files exceeding `max_file_lines` | Refactor large files |
| `orphaned_tests` | Test files not reachable from conftest | Fix test discovery |
| `low_proof_coverage` | Kani coverage <10% on >10 functions | Add proof harnesses |
| `kani_unverified` | >20% of Kani proofs not executed | Run `kani_runner.py --filter not_run` |
| `test_failures` | Test failures in last 24h | Fix failing tests |
| `complexity_increase` | Avg complexity increased >10% | Review recent changes |

### Process Flags
| Flag | Trigger | Action |
|------|---------|--------|
| `blocked_issues` | Any blocked issues exist | Unblock or re-prioritize |
| `blocked_missing_reason` | Blocked issues without `Blocked:` text | Add blocker details |
| `stale_blockers` | Blocked by already-closed issues | Update blocking refs |
| `long_blocked` | Blocked >7 days with no progress | Escalate or unblock |
| `no_work` | No open issues and no in-progress | Create work items |

### System Flags
| Flag | Trigger | Action |
|------|---------|--------|
| `crashes` | Real crashes in last 24h | Check crash logs |
| `untraceable_failures` | Failures with empty working_issues | Review session logs |
| `unclaimed_sessions` | Active sessions but no in-progress issues | Check session status |
| `stuck_process` | Long-running cargo/rustc/z3/etc. | Kill stuck process |

### Git Flags
| Flag | Trigger | Action |
|------|---------|--------|
| `forbidden_ci` | .github/workflows/ files exist | Remove CI files |
| `doc_claim_drift` | Documentation claims don't match code | Update docs |
| `deps_outdated` | Git deps are stale | Run bump_git_dep_rev.sh |
| `git_operation_in_progress` | Incomplete rebase/merge | Abort or complete |
| `repo_dirty_during_pulse` | Files modified during pulse run | Check concurrent sessions |

### Closure Flags
| Flag | Trigger | Action |
|------|---------|--------|
| `high_reopen_rate` | >5% of closed issues reopened | Improve closure quality |

## Configuration

Pulse reads configuration from (first found wins):
1. `pulse.toml`
2. `ai_template_scripts/pulse.toml`
3. `.pulse.toml`

### Example Configuration

```toml
[thresholds]
max_file_lines = 1000       # Flag files exceeding this line count
max_complexity = 15         # Maximum cyclomatic complexity
max_files_over_limit = 5    # Flag when this many files exceed limit
stale_issue_days = 7        # Days before blocked issue is "long blocked"
memory_warning_percent = 80
memory_critical_percent = 90
disk_warning_percent = 80
disk_critical_percent = 90
long_running_process_minutes = 120  # Minutes before process flagged as stuck
large_file_size_gb = 1      # Individual file bloat threshold
tests_dir_size_gb = 10      # tests/ directory bloat threshold
states_dir_size_gb = 10     # TLA+ states/ directory threshold
reports_dir_size_mb = 50    # reports/ directory threshold
git_dep_stale_days = 3      # Days before dep flagged as stale
git_dep_stale_commits = 10  # Commits behind before dep flagged

[large_files]
exclude_patterns = [
    "tests/",
    "crates/star/tests/",
]

[runtime]
skip_orphaned_tests = true  # Skip orphan test detection
```

## Usage

### One-shot Collection (Default)

```bash
# Full metrics with formatted output
./ai_template_scripts/pulse.py

# With progress messages during slow operations
./ai_template_scripts/pulse.py --progress

# Skip GitHub API calls (for rate limit avoidance)
./ai_template_scripts/pulse.py --skip-gh-api

# Fast mode (trades accuracy for speed)
./ai_template_scripts/pulse.py --fast

# Scan git HEAD snapshot for stable metrics
./ai_template_scripts/pulse.py --snapshot
```

### Watch Mode

```bash
# Continuous collection every 5 minutes (default)
./ai_template_scripts/pulse.py --watch

# Custom interval (seconds)
./ai_template_scripts/pulse.py --watch --interval=600
```

### Maintenance

```bash
# Archive old metrics and trim today's file
./ai_template_scripts/pulse.py --rotate

# Convert metrics files to compact JSON
./ai_template_scripts/pulse.py --compact
```

### Broadcast Mode

```bash
# Single-line output for org-wide aggregation
./ai_template_scripts/pulse.py --broadcast
# Output format: repo|branch|head|loc:N|tests:N|proofs:N|issues:N|mem:N%|disk:N%|flags:...
```

## Output Files

### Metrics Directory

Metrics are written to `metrics/`:
- `metrics/YYYY-MM-DD.jsonl` - Daily metrics (one JSON object per line, compact)
- `metrics/latest.json` - Most recent metrics (human-readable, pretty-printed)
- `metrics/archive/` - Compressed old metrics

### Flag Files

Flags are written to `.flags/`:
- Each flag creates a file named after the flag (e.g., `.flags/blocked_issues`)
- Content: timestamp on first line, actionable summary below
- Empty `.flags/` directory means no issues detected

## Monitored Processes

These processes are monitored for long-running detection:
- `cargo`, `rustc` (Rust compilation)
- `cbmc`, `kani` (Rust verification)
- `z3`, `cvc5` (SMT solvers)
- `lean` (Lean prover)

Note: `python` was removed from monitoring in #2139 due to false positives from
daemons (looper.py, tab-title server, memory_watchdog).

Threshold: 120 minutes (configurable via `long_running_process_minutes`)

## Read-Only Invariant

Pulse is strictly read-only. It enforces this by:
1. Capturing dirty file state before collection
2. Comparing after collection
3. Flagging `repo_dirty_during_pulse` if files were modified

This invariant ensures pulse never interferes with concurrent AI sessions.
